# 复杂人机交互链路稳定化改进方案：Harness + WorkflowState + Seed Plan

## Summary

当前项目的核心问题不是单个 Agent 不够聪明，而是协作链路缺少稳定的状态机和执行边界。
改进目标：

```text
聊天事件可路由
用户意图可沉淀
Agent 建议可合并
GM 仲裁可确认
生成计划可增量修改
引擎写入可串行
远程同步可验证
错误/进度/执行状态可见但不暴露内部工程细节
```

最终架构应从“聊天文本驱动一切”升级为：

```text
LANChat Event
-> Interaction Coordinator
-> SessionPlan / SceneSessionState
-> Proposal / Confirmation / Pending Intervention
-> Host Command Queue
-> EngineWriteGate
-> SceneDelta / Actor sync
-> User-facing status
```

## Key Changes

### 1. 用 Harness Coordinator 收口多 Agent

新增或强化一个明确的 `InteractionCoordinator` 概念，职责替代当前散落在 Orchestrator / Worker / HostExecutor 里的隐式判断。

固定分层：

```text
Route Layer:
  判断 @GM / @Agent / 无 @ / message_kind

Intent Layer:
  chat / discussion / planning / generation / edit / conflict / confirmation

Coordinator Layer:
  决定走 RoleAgent、GM summary、GM proposal、pending intervention、host command

Execution Layer:
  只有 confirmed command 进入 Host executor + EngineWriteGate

Sync Layer:
  ActorAdded / ActorTransformUpdated / SceneDelta
```

规则：

- Role Agent 默认只回复和建议。
- GM 默认只总结、仲裁、生成 proposal。
- Host executor 只执行确认后的 command，不执行 discussion。
- 所有引擎写入都走 Host single-writer。

### 2. 引入 Seed Plan，而不是直接让聊天触发生成

为每个房间维护 `SessionPlan`：

```text
scene_goal
style_bible
zones
planned_items
micro_batches
pending_generation_delta
pending_layout_constraints
pending_edit_existing
confirmed_constraints
discussion_summary
```

用户或 Agent 的中途输入不直接改 prompt，而是先写入 plan：

- “后面加灯串和发光蘑菇”
  - 写入 `pending_generation_delta`
- “不要挡中央活动区”
  - 写入 `pending_layout_constraints`
- “先讨论，不要生成”
  - 写入 `mode=DISCUSSING`
- “确认开始”
  - 将 seed plan 变成 executable plan

这样可以解决当前问题：

- 生成中介入不会丢。
- 多 Agent 建议不会互相覆盖。
- prompt 每批从 plan 重建，不靠历史聊天拼接。
- micro-batch 能真实吸收用户新意图。

### 3. 把 WorkflowState 思路移植到 LANChat 长任务

参考 `ai_workflow` 的 `WorkflowState / checkpoint / dialogue_entries`，多人生成应有明确状态：

```text
room_id
session_id
mode: DISCUSSING | PLANNING | CONFIRMING | EXECUTING | PAUSED | FAILED
current_agent
current_batch
total_batches
plan_version
scene_version
pending_proposals
processed_proposals
pending_interventions
last_error
progress_events
```

每个节点只返回 partial update：

```text
intent_parsed
plan_updated
proposal_created
confirmation_received
batch_started
actor_imported
batch_completed
intervention_consumed
final_review_done
```

用户可见状态从这些 events 生成，不直接展示内部函数名。

### 4. Proposal / Command / Status 三类对象分离

当前 proposal 和执行状态混在一起，导致 discussion-only 也进 executor。需要硬分离：

```text
Proposal:
  给用户/房主确认的方案

Command:
  确认后可执行的动作

Status:
  执行队列和结果的可见反馈
```

规则：

- `discussion_only` 不生成 command。
- `gm_summary` 不生成 command。
- `start_generation / actor_add / actor_transform / actor_delete` 才生成 command。
- proposal store 必须支持多 proposal，不再单 pending。
- 每个 command 必须有 `command_id / proposal_id / action_type / expected_result`。

用户可见状态：

```text
已排队
执行中
已完成
未修改场景
执行失败
等待房主确认
```

metadata 才保存：

```text
source_user_id
proposal_id
command_id
execution
scene_version
actor_id
```

### 5. 性能与稳定性分层治理

把“慢”和“卡死”分开处理。

**用户输入链路必须轻：**

- LANChat send 不等待生成。
- Worker 收到消息后先 ack。
- 当前执行 Agent 外的其他 Agent 只轻回复 + 写 pending notes。
- 重任务不占用聊天路由。

**生成链路必须可降级：**

- ResourceSearch 禁止 F5 rebuild。
- image compensation 有总预算，超时跳过。
- 本地模型库优先。
- VLM 默认关闭，第二轮只开 1 target。
- VLM camera 必须离屏，不能动主视口。

**同步链路必须可诊断：**

- ActorAdded 先修 FILE_REQUEST 多根路径映射。
- ActorTransformUpdated 排在 ActorAdded 之后。
- Guest 看不到模型时，第一诊断点是 FILE_REQUEST / FILE_CHUNK，不归因 Agent。

## Implementation Order

### Step 1：交互编排稳定化

- 保持已做的 GM 路由修复。
- 补 `InteractionCoordinator` 或把 Orchestrator 明确拆成 route/intent/decision 三段。
- 普通 `@Agent` discussion 永远不被 GM 抢。
- `@GM 整理/先讨论/暂停/继续` 不进 executor。
- proposal store 和 processed proposal 作为长期状态。

验收：

```text
@商人 围绕暗黑集市主题讨论 -> 商人回复
@GM 整理一下 -> GM summary
@GM 开始执行方案 -> GM proposal
重复确认/拒绝 -> 不重复执行
```

### Step 2：SessionPlan / Pending Intervention 接入 micro-batch

- 当前多人消息进入 `lanchat_scene_runtime` 后，必须进入 SessionPlan。
- `generation_delta` 插入 remaining plan。
- `layout_constraint` 进入下一批 layout context。
- `edit_existing` 已出现 actor 走 fast edit，未出现 actor 等待匹配。
- 每个 micro-batch 开始前 consume pending notes。

验收：

```text
生成前三个物体
中途 @小女孩 后面加灯串
下一批 prompt/context 能看到灯串
最终报告显示已应用/待补/未完成
```

### Step 3：Host Command Queue 收紧

- Host executor 不再执行自然语言 discussion。
- confirmed proposal 转 command list。
- command action type 最小支持：
  - `start_generation`
  - `actor_add`
  - `actor_transform`
  - `actor_delete_or_hide`
  - `apply_plan_delta`
- Command 执行前检查 mode、proposal 状态、scene/session id。
- Command 执行后发 action_status。

验收：

```text
discussion_only -> 未修改场景
start_generation -> 进入 generation queue
actor_transform -> EngineWriteGate
失败 -> 执行失败，不假装完成
```

### Step 4：L4 ActorAdded 同步优先修

- 先修 `FILE_REQUEST` 路径解析。
- `Resource/...`、`assets/local_model_library/...`、运行目录资源都加入 whitelist roots。
- 找不到文件时日志输出 tried roots。
- ActorAdded 成功后再做 transform delta。

验收：

```text
Host 导入 Resource/models/单椅/base.obj
Guest 请求 FILE_REQUEST
Host 发送 FILE_CHUNK
Guest 出现同一 actor
```

### Step 5：Demo 级 F5 脚本固化

主线 demo：

```text
Host 开房
Guest 加入
添加 商人/长者/学者/山贼/GM
@商人 围绕暗黑集市主题讨论
@GM 整理当前想法
房主确认开始生成前三个
生成中 Guest @小女孩 后面加灯串和发光蘑菇
生成中 Guest @学者 后续不要挡中央活动区
@GM 暂停
@GM 继续
后续 batch 吸收 pending notes
Host 移动一个 actor
Guest 看到同步变化
```

判定：

- 如果聊天状态链断：查 Orchestrator / Worker。
- 如果 Host 有 actor、Guest 没有：查 FILE_REQUEST / ActorAdded。
- 如果 Guest 有 actor、移动不同步：查 TransformDelta。
- 如果输入卡死：查 CEF/UI 主线程和长任务阻塞。
- 如果 Agent 错误外露：查 error sanitizer。

## Assumptions

- `deepseed` 未在当前工作区发现，本方案只采用其“seed plan / 先沉淀计划再执行”的架构思想。
- Harness 参考当前 `Agent架构方案.md` 的 Coordinator 模式。
- `ai_workflow` 的 `WorkflowState / checkpoint streaming / global_assets` 是我们可直接借鉴的稳定状态管理方式。
- 本轮目标是 demo 级稳定复杂人机交互，不追求完整 CRDT。
- GM 不提升智能优先级；优先级高于 GM 智能的是状态机、proposal 生命周期、pending plan、L4 同步、性能降级。
