## 零、架构分层（共识地基）

```text
Generation Layer   并行：模型生成、LLM/VLM 调用（纯 API，不碰引擎）
Planning Layer     异步：布局计划、校验计划、依赖判断
```

## 零·二、核心数据模型（四个对象）—— 下阶段的真正主线

> **阶段定性（2026-06-13）**：下阶段不再堆功能。目标是把当前"能生成场景"升级成：
> **可反悔、可渐进、可进入、可保护用户意图的开放场景生成系统。**
>
> 实现这一步的前提是：**不要再把"当前场景"当成唯一对象。** 否则重新生成、用户反悔、历史模型复用都无从下手。正式引入四个核心对象，各自独立的生命周期：

```text
AssetPool    资产池：所有生成/导入过的模型，不随重新生成删除（只增）
SceneLayout  当前布局：哪些资产被放进场景、各自的 pos/rot/scale（资产实例引用）
UserLayer    用户操作账本：锁定集合 + 拒绝集合 + 操作日志 + 偏好（不等于 provenance）
ZoneTree     空间树：开放场景的通用递归空间结构（见任务 C）
```

| 对象 | 生命周期 | regenerate 时 | 谁能删 |
|------|---------|--------------|--------|
| AssetPool | 本地持久（磁盘 .glb + metadata.json），可选本地向量索引，几乎只增 | **完全不动** | 几乎不删 |
| SceneLayout | 一次布局的快照（资产实例引用 + transform） | Agent 摆放可清 | regenerate |
| UserLayer | 跟随用户操作（账本） | **保留** | 仅用户显式删 |
| ZoneTree | 一次场景的空间规划 | 重新规划 | regenerate |

> **AssetPool 默认不依赖向量库**（资产只能本地、不上传线上）。最小版就是 `models/<name>/base.glb + metadata.json`，"历史模型可 re-add" 只需列目录。本地向量索引是**可选增强**（语义检索"再来个类似的"），不是默认范围，避免实现膨胀。
>
> **UserLayer ≠ provenance**（G老师修正）。provenance 是挂在每个资产实例上的 actor metadata（它是谁的、锁没锁）；UserLayer 是**全局的用户操作账本**——记录用户为什么碰、什么时候碰、表达了什么偏好。prompt 懒重建要的是后者：只看 provenance 只能知道"这个被碰过"，不知道"用户想要什么"。

### 修订后的核心数据模型（G老师审查定稿）

```python
Asset {                    # AssetPool 的元素：一个生成/导入过的模型
  asset_id
  local_path
  prompt                   # 当初用什么 prompt 生成的（支持"再来个类似的"）
  type
  created_at
  status                   # available | hidden | rejected
}

LayoutInstance {           # SceneLayout 的元素：一个资产在场景里的一次摆放
  instance_id
  asset_id                 # 引用 AssetPool，不复制
  zone_id                  # 挂在哪个 Zone 上（位置/约束都相对 Zone）
  transform                # pos / rot / scale
  provenance               # "USER" | "AGENT"
  owner_id                 # peer_id | "local"（多人：谁拥有/锁定）
  lock_level               # "NONE" | "SOFT" | "HARD"
  touched_by_user          # bool
  anchor_ref               # str | None（依赖的锚点实例）
  batch_id                 # str | None（哪一轮生成的，settlement 缩范围用）
  layout_status            # "active" | "hidden" | "rejected" | "stale"
}

UserLayer {                # 全局用户操作账本（不是 per-actor）
  locked_instance_ids
  rejected_asset_ids
  operation_log
  preferences
}
```

> **七字段 provenance**（G老师从五字段扩展）：加 `owner_id`（多人锁定可见性）+ `layout_status`（用户拒绝 / 历史可恢复 / 失效需重算）。没有这两个，"用户拒绝物体""历史模型可恢复""多人锁定"都缺状态表达。`owner_id` 这点与下方「同传约束 3」是同一个洞——一开始就带 owner 维度，否则联机返工。

### 总原则（开放场景生成的定义）

> **开放场景生成不是靠枚举场景类型，而是靠 LLM 将任意需求分解成 `AssetPool + ZoneTree + SceneLayout`；代码只执行通用结构，不写 `if 教堂 / if 蒙古包 / if 客厅`。**

### regenerate 语义（必须先定清楚）

```text
重新生成 ≠ 删除旧模型
重新生成 = 清除 Agent 自动布局 + 保留用户锁定物体 + 保留资产池 + 重新规划场景
```

| 类型 | regenerate 时处理 |
|------|------------------|
| AssetPool 资产 | 不删除 |
| Agent 自动摆放 | 可清除 |
| 用户手动加入 | 默认保留 |
| 用户锁定物体 | 必须保留 |
| 用户拒绝物体 | 从布局移除，但资产池保留 |
| 新一轮生成资产 | 加入 AssetPool |

目标用户故事："重新生成一版，但保留这个蒙古包和那匹马。" —— 系统必须能做到。

### ⚠️ 与聊天室同传的关系（代码核查后的真实约束）

四层模型在**多人联机**下不是单机的简单推广，有三条来自现有同传机制的硬约束：

1. **同传是 actor 级广播，不是 layout 级。** [actor.py:208](editor/CoronaCore/core/entities/actor.py#L208) `_broadcast_actor_created` 在每个 actor 创建时发 `actor-sync-broadcast`（`to_dict()` = 路径+变换）。**没有"整份 SceneLayout 作为单元同步"的概念**——每个对端是从 actor 事件流里各自重建布局。后果：**regenerate = 海量 remove + 海量 re-add = 一阵同步事件风暴**。这条让 `_engine_lock` 串行收口 + 防抖在多人下从"锦上添花"变成"必需"。

2. **AssetPool 是每机私有，不自动共享。** [actor.py:228-264](editor/CoronaCore/core/entities/actor.py#L228-L264) `_ensure_network_model_path_in_project`：只有 actor **被放进场景**时，才把 .glb 拷进项目 `Resource/` 并随广播传给对端。所以对端拿到的是**已放置**的资产；A 生成但**未放置**的模型不在 B 的池里。"历史模型可 re-add"面板默认只能列**本机**池——A 和 B 的 AssetPool 不一致。若要共享未放置资产，需显式同步机制（不在当前同传链路里）。

3. **provenance `touched_by_user: bool` 在多人下太粗。** 联机有多个用户，A 的锁应对 B 可见。该字段需升级为 `touched_by: <peer_id>`，否则无法表达"这是张三锁的，李四别动"。[actor.py:46-49](editor/CoronaCore/core/entities/actor.py#L46-L49) 已有 `network_remote` 标志区分本地/远端 actor，可作为 owner 维度的起点。

> 结论：四层模型在单机先立起来，但 **UserLayer 的 provenance 从一开始就该带 owner（peer_id）维度**，否则联机时要返工。regenerate 在多人下"清谁的布局"也是真歧义——默认只清发起者自己的 Agent 摆放，别动别人的。

## 任务 A：主链路稳定性

目标：`并行生成 → 组装导入 → 场景可用`，且不丢同事反馈的"旋转角"痛点。

### 开放式意图路由（方向决策，2026-06-12）

> **背景**：原路由（同事 fox-zq `77137784 第一波接聊天室测试` 初版）靠两道写死的关键词门——`_SCENE_COMMAND_PATTERNS` + `is_compose_request` 的房间类型清单。清单外的场景描述（"赛博朋克酒吧街""海底世界""蒙古包草原"=Task C 目标场景）全部掉进 chat，与"开放式场景生成"的设想从架构上矛盾。补词只是创可贴，无法根治。

**决策：关键词门未命中时，改用一次轻量 LLM 调用判别"是否要生成完整场景"**（不再靠词表）。

- 关键词命中 → 快路径，零额外 LLM 延迟（保留）
- 关键词漏判 → `is_compose_intent()` 用 LLM 语义判断，覆盖任意场景描述（开放式）
- 明显闲聊（`_NON_SCENE_PATTERNS`）→ 不调 LLM，直接 chat
- LLM 不可用 → 保守退回旧关键词行为（宁漏判不误触发）

这让"用户说什么场景都能造"成立，是开放式生成的入口前提。详见 [实施修改记录.md](实施修改记录.md) A-1.6。

### P0/P1 旋转兜底（独立补充，关 review 的代价对冲）

关掉 review 等于关掉唯一处理旋转的机制。分两层兜底，**prompt 先上（零成本、当轮生效），函数后补（更硬、LLM 失效也保底）**：

**P0 — prompt 层**：按物体语义给默认朝向规则，注入 compose_scene prompt：

```text
布局规则（注入 compose_scene prompt）：
- 座椅类（椅/凳/沙发）：正面朝向最近的桌/茶几/电视，背靠墙
- 床头柜/边几：长边贴靠主体（床/沙发）
- 灯具（台灯/落地灯）：直立，灯头朝上
- 柜体（衣柜/电视柜/书架）：背面贴墙，开口朝房间内
- 桌类：长边平行于最近的墙
```

**P1 — 函数层**：prompt 失效或 LLM 输出明显异常时的工程兜底。它需要的 `nearest_wall` / `anchor_actor` 正是 Constraint Solver 已经算出的数据，直接复用：

```python
def infer_default_rotation(actor_type, nearest_wall, anchor_actor):
    # chair/sofa  → 朝向 anchor（桌/电视）
    # cabinet/bookshelf → 背靠最近墙
    # lamp        → upright（不旋转）
    # table       → 长边平行最近墙
    ...
```

prompt 不碰引擎、不截图，先把旋转问题压到可接受范围；函数层在布局阶段对缺失/异常旋转做最后一道补齐。等 review 修好接回再做精修。

## 任务 B：渐进式生成 + 用户介入

目标：模型逐步出现，用户随时增删改，系统尊重用户操作，后续生成/布局不覆盖用户意图。

### ⚠️ 三条前置硬伤（代码核查后确认，不解则问题①做不出真渐进）

核查结论：四个问题在计划里都有对应条目，问题②③④覆盖扎实。但**问题①（渐进 + 介入 + 即时反馈）有三处当前架构层面的硬伤，原计划没点破，其中第 1 条与任务 A "保留原管线" 直接矛盾**。这三条是问题①的真正地基，必须前置，不能放 P3。

**硬伤 1：渐进式导入 与 复用 `import_to_engine_node` 直接冲突** 🔴

[import_to_engine.py:69](editor/plugins/AITool/cai_extensions/flows/scene_composition_workflow/import_to_engine.py#L69) 开头就"清除上一轮 N 个 actor"。每批渐进导入若复用它，会把上一批 + 用户刚摆的东西全抹掉。
→ 必须新开一条**增量导入路径**：只 `add` 不 `clear`，且按 provenance 跳过 USER 物体。这是渐进式的**前置地基**，不是 P3。任务 A 的"保留 `import_to_engine_node`"只对**非渐进**主链路成立；渐进路径走新入口。

**硬伤 2：用户介入的输入通道未定义** 🔴

`touched_by_user` 要跟踪用户改动，但**用户怎么介入**没定义，两条来源处置不同：
- 走 chat 指令（"把椅子放大"）→ 经 coordinator，**当前已能捕获**，直接标 `touched_by_user`。
- 走引擎视口直接拖拽/缩放 → 引擎侧无回调进 Python，**当前无感知**。只能靠每轮 `get_actors()`（[agent_adapter.py:711](editor/plugins/AITool/cai_extensions/agent/agent_adapter.py#L711) 已在注入）快照 + **diff 上一帧**找出"谁被人动过"。
→ 需补一个 **scene-diff 器**（对比 transform/material 快照），作为 `touched_by_user` 的隐含前置。用户原话强调"随机性操作"+"即时反馈"，若含视口操作则此器必需。

**硬伤 3：`compose()` 是单次阻塞调用，不可中断** 🔴（最深）

"边生成边让用户随机介入并即时反馈" = **Agent 生成循环与用户操作并发**。但当前是**回合制**：一条 chat → 一次完整 [compose()](editor/plugins/AITool/cai_extensions/agent/scene_composer.py#L326) 阻塞跑完，没有"后台逐批生成 + 用户同时操作"的概念。
→ `_engine_lock` 是给并发写加锁，但**前提是先有并发**。要有并发，`compose()` 得改成可在 tier/batch 间 **yield、回读场景状态、被打断**的分批循环（或事件循环）。这比 debounce（B-P2 擦边提到）更底层，是 `_engine_lock` 能发挥作用的前提。

> 这三条不解，问题① 只能做出"看起来渐进、实则仍是批处理"的效果。下表已把**增量导入路径**和 **compose 可中断化**提为前置层（B-P-1 / B-P0），与 provenance 验证并列。

### 共识：先做最小机制，不做完整 Command Bus

`_engine_lock + provenance + 防抖 + 用户保护` —— 一把锁 + 四字段拿到 Workshop 需要的全部能力，等真出现高并发争抢痛点再升级为带优先级的调度器。

### provenance 载体：默认旁路字典（方案 B）

引擎 actor 是 C++ 对象，生命周期 / 序列化 / 复制 / 重载后极可能丢 Python 动态属性，不赌"能挂上"。**默认走旁路 `SceneState` 字典**，所有 provenance/lock/anchor 信息存这里，actor 只用稳定 id 索引：

```python
scene_actor_meta = {
    actor_id: {
        "provenance": "USER" | "AGENT",
        "lock_level": "NONE" | "SOFT" | "HARD",
        "touched_by_user": bool,
        "anchor_ref": str | None,
        "batch_id": str | None,   # 本次生成批次，settlement 缩范围用
    }
}
```

> 比原稿增一个字段 `batch_id`：让 settlement 缩范围从"本批新导入"这种模糊表述变成可判定条件 `batch_id == current_batch`。
>
> **仍保留一个 5 分钟探针**：确认 actor 是否有稳定可索引的 id（id 不稳则字典 key 失效），以及方案 A 是否意外可用（可用则省一层间接）。但默认实现按方案 B 推进，不等探针结论卡住 P0。

## 任务 C：通用空间模型（递归 Zone）—— 开放场景生成的上位抽象

> **方向修正（2026-06-13）**：原计划把"mixed environment"当作 P3 末端的特例。错了。**它是统领全局的通用模型，现在的 `_generate_room_box` 反而是它的退化特例。** 出发点是「开放场景生成」——不枚举场景类型（教堂走A、蒙古包走B、客厅走C），而是用一个递归结构表达任意空间，判断逻辑长在 LLM 分解那端，代码只忠实执行。

### 核心抽象：递归的「空间区域」(Zone)

```python
Zone {
  id
  name
  role            # 这个区域的语义角色（室外/室内/连接区…）
  volume          # 参数化空间体积 (bounds) —— 唯一事实源，我们自己造、可控、能留门洞
  enclosure       # "none" | "terrain" | "box" | "shell" —— 这个体积"怎么被包裹"
  primary_shell_asset_id   # 主外壳：从外面看的主体建筑（蒙古包帐篷/教堂立面）
  dressing_assets          # [asset_id] —— 附加外部装饰，一个 Zone 可有多个
  interior_skin            # 参数化内皮（见下，不是模型）：进去后墙地顶
  objects         # [Object] —— 放在这个体积里的物体
  sub_zones       # [Zone] —— 递归：区域里可嵌套区域
  connectors      # [Connector] —— 门洞/通道，连接父子 Zone，camera 由此穿越
}
```

> **外壳必须支持多个 dressing（G老师修正）**：一个 Zone 的外部往往不止一个模型——教堂有墙体/塔楼/门/彩窗，蒙古包有帐篷外壳/门帘/烟囱，营地有围栏/地形/道路。所以 `dressing_assets: [asset_id]` 是列表，`primary_shell_asset_id` 指主体。最小版可以只有一个，但数据模型从一开始就支持多个，避免后期返工。
>
> **interior_skin 是参数化结构，不是生成模型（G老师修正，关键）**：内部可见面**必须由参数化 skin 生成，不靠外壳模型的内表面**（文生3D 团块没有可用内表面）。

```python
interior_skin = {
  "floor":    ...,    # 地面材质/类型
  "wall":     ...,    # 墙面
  "ceiling":  ...,    # 天花/顶
  "openings": [...],  # 开口（门/窗），与 connectors 对齐
}
```

蒙古包：`{floor: "carpet", wall: "felt_with_wood_frame", ceiling: "dome_liner", openings: ["door"]}`
教堂：`{floor: "stone", wall: "stone_wall", ceiling: "vault", openings: ["door", "stained_glass_window"]}`

> **为什么 `shell` 和 `interior_skin` 必须分开**：
> - `primary_shell_asset_id` + `dressing_assets` 负责**外面看起来像什么**（生成模型，缩放包住体积）
> - `interior_skin` 负责**进去后墙地顶是什么**（参数化贴皮，由代码按体积生成可见内表面）
>
> 现在 box 的六面白墙正是 `interior_skin` 的最朴素取值（floor/wall/ceiling 全是默认中性面）。这比"把教堂塞进中性 box"或"指望外壳模型有内表面"都合理。

任何场景都是一棵 Zone 树，**没有特例**：

```text
草原上的蒙古包：
Zone 草原                enclosure=terrain（开放，无墙无顶）
├─ 栅栏 / 马 / 篝火       objects
└─ Zone 蒙古包内部        enclosure=box（带门洞）+ 蒙古包外壳 dressing
   ├─ 地毯 / 矮桌 / 坐垫   objects
   └─ (可继续嵌套)

欧式教堂：  Zone 教堂  enclosure=box(放大,带门) + 教堂外壳   objects=祭坛/长椅/立柱
现代客厅：  Zone 客厅  enclosure=box                       objects=沙发/茶几/电视
            └─ 退化情况：单 Zone、无 sub_zones、enclosure=box —— 不是"第三类"，是树的最简形态
```

### 关键约束：volume 是事实源，生成的建筑只是「外壳 dressing」

必须诚实承认的物理限制：**文生3D 给的"教堂""蒙古包"是实心外观团块，不是能走进去的空心内部。** 所以：

- 每个 Zone 的**内部体积永远是参数化定义的**（我们自己造的盒子，可控、能开门洞）
- 生成的建筑模型只是**套在体积外面的视觉外壳 (dressing)**，缩放到包住体积、门洞对齐
- 物体永远放进参数化体积里，由 Constraint Solver 摆放；外壳只负责"从外面看像个蒙古包/教堂"

| 场景 | volume（事实源） | enclosure dressing |
|------|----------------|-------------------|
| 客厅 | 房间尺寸 | 合成盒子（= 现在的 `_generate_room_box`）|
| 蒙古包内部 | 内部空间 | 生成的蒙古包外壳，缩放包住体积 + 门洞对齐 |
| 草原 | 一大片区域 | terrain 铺底，无墙无顶 |
| 教堂 | 放大的体积 | 生成的教堂外壳，或合成盒子放大 |

`_generate_room_box` 不被取代，它就是 `enclosure=box` 这一种 dressing 的实现——从"无条件对所有室内生成"变成"Zone 树里某节点声明了 box 才生成"。

### 为什么这才是"开放"的

**开放性不在代码，在分解那一步。** 一个 LLM 把任意 prompt 读成一棵 Zone 树，代码不枚举任何场景类型，只递归实例化 `(volume + enclosure + objects + sub_zones)`。"草原上的蒙古包""赛博朋克酒吧地下室""欧式教堂"对代码是同一件事。我们不写 `if 教堂`，我们让分解器输出 `enclosure=box, scale=large`。

camera 从草原走进蒙古包 = 穿过一个 Zone 边界。没有 Portal / LOD / 动态透明 / 双渲染——蒙古包那面墙几何上留了门洞，就这么简单。**否定一切引擎级渲染技巧，纯几何。**

### 复用已有工程，不另起炉灶

- [multi_scene_parallel_workflow](editor/plugins/AITool/cai_extensions/flows/multi_scene_parallel_workflow/) 的 decompose 已在"把需求拆成子场景"——Zone 树是把那个**扁平的并行子场景**升级成**带空间嵌套 + enclosure 类型的树**。
- `terrain_generation_workflow` 已接入 = `enclosure=terrain` 的现成实现。
- `camera_move` 已验证 = 穿门洞的现成手段。

### 落地分层（Workshop 周期内务实推进）

概念模型立成 general，落地按深度分层，每层都能独立验证：

1. **数据结构定成 Zone**（`enclosure` 是字段不是 if 分支）。初版只支持**单 Zone + box**——把现在的代码"重新表达"成 Zone 退化形态，零功能损失但骨架对了。
2. **支持两层嵌套**（一个外 Zone + 一个内 Zone + 门洞）——足够 demo 蒙古包草原。
3. **shell dressing**（生成的建筑外壳缩放包住体积 + 门洞对齐）、**terrain 与 box 融合**。

工作量集中在：`_detect_scene_indoor` 从二值 bool 升级为"分解出 Zone 树"+ 盒子开洞 + 体积内放置 + 外壳对齐。
