## 零、架构分层（共识地基）

```text
Generation Layer   并行：模型生成、LLM/VLM 调用（纯 API，不碰引擎）
```

## 零·二、核心数据模型（四个对象）—— 下阶段的真正主线

> **阶段定性（2026-06-13）**：下阶段不再堆功能。目标是把当前"能生成场景"升级成：
> **可反悔、可渐进、可进入、可保护用户意图的开放场景生成系统。**
>
> 实现这一步的前提是：**不要再把"当前场景"当成唯一对象。** 否则重新生成、用户反悔、历史模型复用都无从下手。正式引入四个核心对象，各自独立的生命周期：

```text
AssetPool    资产池：所有生成/导入过的模型，不随重新生成删除（只增）
SceneLayout  当前布局：哪些资产被放进场景、各自的 pos/rot/scale（资产实例引用）
UserLayer    用户操作账本：锁定集合 + 拒绝集合 + 操作日志 + 偏好（不等于 provenance）
ZoneTree     空间树：开放场景的通用递归空间结构（见任务 C）
```

| 对象 | 生命周期 | regenerate 时 | 谁能删 |
|------|---------|--------------|--------|
| AssetPool | 本地持久（磁盘 .glb + metadata.json），可选本地向量索引，几乎只增 | **完全不动** | 几乎不删 |
| SceneLayout | 一次布局的快照（资产实例引用 + transform） | Agent 摆放可清 | regenerate |
| UserLayer | 跟随用户操作（账本） | **保留** | 仅用户显式删 |
| ZoneTree | 一次场景的空间规划 | 重新规划 | regenerate |

> **AssetPool 默认不依赖向量库**（资产只能本地、不上传线上）。最小版就是 `models/<name>/base.glb + metadata.json`，"历史模型可 re-add" 只需列目录。本地向量索引是**可选增强**（语义检索"再来个类似的"），不是默认范围，避免实现膨胀。
>
> **UserLayer ≠ provenance**（G老师修正）。provenance 是挂在每个资产实例上的 actor metadata（它是谁的、锁没锁）；UserLayer 是**全局的用户操作账本**——记录用户为什么碰、什么时候碰、表达了什么偏好。prompt 懒重建要的是后者：只看 provenance 只能知道"这个被碰过"，不知道"用户想要什么"。

### 修订后的核心数据模型（G老师审查定稿）

```python
Asset {                    # AssetPool 的元素：一个生成/导入过的模型
  asset_id
  local_path
  prompt                   # 当初用什么 prompt 生成的（支持"再来个类似的"）
  type
  created_at
  status                   # available | hidden | rejected
}

LayoutInstance {           # SceneLayout 的元素：一个资产在场景里的一次摆放
  instance_id
  asset_id                 # 引用 AssetPool，不复制
  zone_id                  # 挂在哪个 Zone 上（位置/约束都相对 Zone）
  transform                # pos / rot / scale
  provenance               # "USER" | "AGENT"
  owner_id                 # peer_id | "local"（多人：谁拥有/锁定）
  lock_level               # "NONE" | "SOFT" | "HARD"
  touched_by_user          # bool
  anchor_ref               # str | None（依赖的锚点实例）
  batch_id                 # str | None（哪一轮生成的，settlement 缩范围用）
  layout_status            # "active" | "hidden" | "rejected" | "stale"
}

UserLayer {                # 全局用户操作账本（不是 per-actor）
  locked_instance_ids
  rejected_asset_ids
  operation_log
  preferences
}
```

> **七字段 provenance**（G老师从五字段扩展）：加 `owner_id`（多人锁定可见性）+ `layout_status`（用户拒绝 / 历史可恢复 / 失效需重算）。没有这两个，"用户拒绝物体""历史模型可恢复""多人锁定"都缺状态表达。`owner_id` 这点与下方「同传约束 3」是同一个洞——一开始就带 owner 维度，否则联机返工。

### 总原则（开放场景生成的定义）

> **开放场景生成不是靠枚举场景类型，而是靠 LLM 将任意需求分解成 `AssetPool + ZoneTree + SceneLayout`；代码只执行通用结构，不写 `if 教堂 / if 蒙古包 / if 客厅`。**

### regenerate 语义（必须先定清楚）

```text
重新生成 ≠ 删除旧模型
重新生成 = 清除 Agent 自动布局 + 保留用户锁定物体 + 保留资产池 + 重新规划场景
```

| 类型 | regenerate 时处理 |
|------|------------------|
| AssetPool 资产 | 不删除 |
| Agent 自动摆放 | 可清除 |
| 用户手动加入 | 默认保留 |
| 用户锁定物体 | 必须保留 |
| 用户拒绝物体 | 从布局移除，但资产池保留 |
| 新一轮生成资产 | 加入 AssetPool |

目标用户故事："重新生成一版，但保留这个蒙古包和那匹马。" —— 系统必须能做到。

### ⚠️ 与聊天室同传的关系（代码核查后的真实约束）

四层模型在**多人联机**下不是单机的简单推广，有三条来自现有同传机制的硬约束：

1. **同传是 actor 级广播，不是 layout 级。** [actor.py:208](editor/CoronaCore/core/entities/actor.py#L208) `_broadcast_actor_created` 在每个 actor 创建时发 `actor-sync-broadcast`（`to_dict()` = 路径+变换）。**没有"整份 SceneLayout 作为单元同步"的概念**——每个对端是从 actor 事件流里各自重建布局。后果：**regenerate = 海量 remove + 海量 re-add = 一阵同步事件风暴**。这条让 `_engine_lock` 串行收口 + 防抖在多人下从"锦上添花"变成"必需"。

2. **AssetPool 是每机私有，不自动共享。** [actor.py:228-264](editor/CoronaCore/core/entities/actor.py#L228-L264) `_ensure_network_model_path_in_project`：只有 actor **被放进场景**时，才把 .glb 拷进项目 `Resource/` 并随广播传给对端。所以对端拿到的是**已放置**的资产；A 生成但**未放置**的模型不在 B 的池里。"历史模型可 re-add"面板默认只能列**本机**池——A 和 B 的 AssetPool 不一致。若要共享未放置资产，需显式同步机制（不在当前同传链路里）。

3. **provenance `touched_by_user: bool` 在多人下太粗。** 联机有多个用户，A 的锁应对 B 可见。该字段需升级为 `touched_by: <peer_id>`，否则无法表达"这是张三锁的，李四别动"。[actor.py:46-49](editor/CoronaCore/core/entities/actor.py#L46-L49) 已有 `network_remote` 标志区分本地/远端 actor，可作为 owner 维度的起点。

> 结论：四层模型在单机先立起来，但 **UserLayer 的 provenance 从一开始就该带 owner（peer_id）维度**，否则联机时要返工。regenerate 在多人下"清谁的布局"也是真歧义——默认只清发起者自己的 Agent 摆放，别动别人的。

## 任务 A：主链路稳定性

目标：`并行生成 → 组装导入 → 场景可用`，且不丢同事反馈的"旋转角"痛点。

### 开放式意图路由（方向决策，2026-06-12）

> **背景**：原路由（同事 fox-zq `77137784 第一波接聊天室测试` 初版）靠两道写死的关键词门——`_SCENE_COMMAND_PATTERNS` + `is_compose_request` 的房间类型清单。清单外的场景描述（"赛博朋克酒吧街""海底世界""蒙古包草原"=Task C 目标场景）全部掉进 chat，与"开放式场景生成"的设想从架构上矛盾。补词只是创可贴，无法根治。

**决策：关键词门未命中时，改用一次轻量 LLM 调用判别"是否要生成完整场景"**（不再靠词表）。

- 关键词命中 → 快路径，零额外 LLM 延迟（保留）
- 关键词漏判 → `is_compose_intent()` 用 LLM 语义判断，覆盖任意场景描述（开放式）
- 明显闲聊（`_NON_SCENE_PATTERNS`）→ 不调 LLM，直接 chat
- LLM 不可用 → 保守退回旧关键词行为（宁漏判不误触发）

这让"用户说什么场景都能造"成立，是开放式生成的入口前提。详见 [实施修改记录.md](实施修改记录.md) A-1.6。

### P0/P1 旋转兜底（独立补充，关 review 的代价对冲）

关掉 review 等于关掉唯一处理旋转的机制。分两层兜底，**prompt 先上（零成本、当轮生效），函数后补（更硬、LLM 失效也保底）**：

**P0 — prompt 层**：按物体语义给默认朝向规则，注入 compose_scene prompt：

```text
布局规则（注入 compose_scene prompt）：
- 座椅类（椅/凳/沙发）：正面朝向最近的桌/茶几/电视，背靠墙
- 床头柜/边几：长边贴靠主体（床/沙发）
- 灯具（台灯/落地灯）：直立，灯头朝上
- 柜体（衣柜/电视柜/书架）：背面贴墙，开口朝房间内
- 桌类：长边平行于最近的墙
```

**P1 — 函数层**：prompt 失效或 LLM 输出明显异常时的工程兜底。它需要的 `nearest_wall` / `anchor_actor` 正是 Constraint Solver 已经算出的数据，直接复用：

```python
def infer_default_rotation(actor_type, nearest_wall, anchor_actor):
    # chair/sofa  → 朝向 anchor（桌/电视）
    # cabinet/bookshelf → 背靠最近墙
    # lamp        → upright（不旋转）
    # table       → 长边平行最近墙
    ...
```

prompt 不碰引擎、不截图，先把旋转问题压到可接受范围；函数层在布局阶段对缺失/异常旋转做最后一道补齐。等 review 修好接回再做精修。

## 任务 B：渐进式生成 + 用户介入

目标：模型逐步出现，用户随时增删改，系统尊重用户操作，后续生成/布局不覆盖用户意图。

### ⚠️ 三条前置硬伤（代码核查后确认，不解则问题①做不出真渐进）

核查结论：四个问题在计划里都有对应条目，问题②③④覆盖扎实。但**问题①（渐进 + 介入 + 即时反馈）有三处当前架构层面的硬伤，原计划没点破，其中第 1 条与任务 A "保留原管线" 直接矛盾**。这三条是问题①的真正地基，必须前置，不能放 P3。

**硬伤 1：渐进式导入 与 复用 `import_to_engine_node` 直接冲突** 🔴

[import_to_engine.py:69](editor/plugins/AITool/cai_extensions/flows/scene_composition_workflow/import_to_engine.py#L69) 开头就"清除上一轮 N 个 actor"。每批渐进导入若复用它，会把上一批 + 用户刚摆的东西全抹掉。
→ 必须新开一条**增量导入路径**：只 `add` 不 `clear`，且按 provenance 跳过 USER 物体。这是渐进式的**前置地基**，不是 P3。任务 A 的"保留 `import_to_engine_node`"只对**非渐进**主链路成立；渐进路径走新入口。

**硬伤 2：用户介入的输入通道未定义** 🔴

`touched_by_user` 要跟踪用户改动，但**用户怎么介入**没定义，两条来源处置不同：
- 走 chat 指令（"把椅子放大"）→ 经 coordinator，**当前已能捕获**，直接标 `touched_by_user`。
- 走引擎视口直接拖拽/缩放 → 引擎侧无回调进 Python，**当前无感知**。只能靠每轮 `get_actors()`（[agent_adapter.py:711](editor/plugins/AITool/cai_extensions/agent/agent_adapter.py#L711) 已在注入）快照 + **diff 上一帧**找出"谁被人动过"。
→ 需补一个 **scene-diff 器**（对比 transform/material 快照），作为 `touched_by_user` 的隐含前置。用户原话强调"随机性操作"+"即时反馈"，若含视口操作则此器必需。

**硬伤 3：`compose()` 是单次阻塞调用，不可中断** 🔴（最深）

"边生成边让用户随机介入并即时反馈" = **Agent 生成循环与用户操作并发**。但当前是**回合制**：一条 chat → 一次完整 [compose()](editor/plugins/AITool/cai_extensions/agent/scene_composer.py#L326) 阻塞跑完，没有"后台逐批生成 + 用户同时操作"的概念。
→ `_engine_lock` 是给并发写加锁，但**前提是先有并发**。要有并发，`compose()` 得改成可在 tier/batch 间 **yield、回读场景状态、被打断**的分批循环（或事件循环）。这比 debounce（B-P2 擦边提到）更底层，是 `_engine_lock` 能发挥作用的前提。

> 这三条不解，问题① 只能做出"看起来渐进、实则仍是批处理"的效果。下表已把**增量导入路径**和 **compose 可中断化**提为前置层（B-P-1 / B-P0），与 provenance 验证并列。

### 共识：先做最小机制，不做完整 Command Bus

`_engine_lock + provenance + 防抖 + 用户保护` —— 一把锁 + 四字段拿到 Workshop 需要的全部能力，等真出现高并发争抢痛点再升级为带优先级的调度器。

### provenance 载体：默认旁路字典（方案 B）

引擎 actor 是 C++ 对象，生命周期 / 序列化 / 复制 / 重载后极可能丢 Python 动态属性，不赌"能挂上"。**默认走旁路 `SceneState` 字典**，所有 provenance/lock/anchor 信息存这里，actor 只用稳定 id 索引：

```python
scene_actor_meta = {
    actor_id: {
        "provenance": "USER" | "AGENT",
        "lock_level": "NONE" | "SOFT" | "HARD",
        "touched_by_user": bool,
        "anchor_ref": str | None,
        "batch_id": str | None,   # 本次生成批次，settlement 缩范围用
    }
}
```

> 比原稿增一个字段 `batch_id`：让 settlement 缩范围从"本批新导入"这种模糊表述变成可判定条件 `batch_id == current_batch`。
>
> **仍保留一个 5 分钟探针**：确认 actor 是否有稳定可索引的 id（id 不稳则字典 key 失效），以及方案 A 是否意外可用（可用则省一层间接）。但默认实现按方案 B 推进，不等探针结论卡住 P0。

## 任务 C：通用空间模型（递归 Zone）—— 开放场景生成的上位抽象

> **方向修正（2026-06-13）**：原计划把"mixed environment"当作 P3 末端的特例。错了。**它是统领全局的通用模型，现在的 `_generate_room_box` 反而是它的退化特例。** 出发点是「开放场景生成」——不枚举场景类型（教堂走A、蒙古包走B、客厅走C），而是用一个递归结构表达任意空间，判断逻辑长在 LLM 分解那端，代码只忠实执行。

### 核心抽象：递归的「空间区域」(Zone)

```python
Zone {
  id
  name
  role            # 这个区域的语义角色（室外/室内/连接区…）
  volume          # 参数化空间体积 (bounds) —— 唯一事实源，我们自己造、可控、能留门洞
  enclosure       # "none" | "terrain" | "box" | "shell" —— 这个体积"怎么被包裹"
  primary_shell_asset_id   # 主外壳：从外面看的主体建筑（蒙古包帐篷/教堂立面）
  dressing_assets          # [asset_id] —— 附加外部装饰，一个 Zone 可有多个
  interior_skin            # 参数化内皮（见下，不是模型）：进去后墙地顶
  objects         # [Object] —— 放在这个体积里的物体
  sub_zones       # [Zone] —— 递归：区域里可嵌套区域
  connectors      # [Connector] —— 门洞/通道，连接父子 Zone，camera 由此穿越
}
```

> **外壳必须支持多个 dressing（G老师修正）**：一个 Zone 的外部往往不止一个模型——教堂有墙体/塔楼/门/彩窗，蒙古包有帐篷外壳/门帘/烟囱，营地有围栏/地形/道路。所以 `dressing_assets: [asset_id]` 是列表，`primary_shell_asset_id` 指主体。最小版可以只有一个，但数据模型从一开始就支持多个，避免后期返工。
>
> **interior_skin 是参数化结构，不是生成模型（G老师修正，关键）**：内部可见面**必须由参数化 skin 生成，不靠外壳模型的内表面**（文生3D 团块没有可用内表面）。

```python
interior_skin = {
  "floor":    ...,    # 地面材质/类型
  "wall":     ...,    # 墙面
  "ceiling":  ...,    # 天花/顶
  "openings": [...],  # 开口（门/窗），与 connectors 对齐
}
```

蒙古包：`{floor: "carpet", wall: "felt_with_wood_frame", ceiling: "dome_liner", openings: ["door"]}`
教堂：`{floor: "stone", wall: "stone_wall", ceiling: "vault", openings: ["door", "stained_glass_window"]}`

> **为什么 `shell` 和 `interior_skin` 必须分开**：
> - `primary_shell_asset_id` + `dressing_assets` 负责**外面看起来像什么**（生成模型，缩放包住体积）
> - `interior_skin` 负责**进去后墙地顶是什么**（参数化贴皮，由代码按体积生成可见内表面）
>
> 现在 box 的六面白墙正是 `interior_skin` 的最朴素取值（floor/wall/ceiling 全是默认中性面）。这比"把教堂塞进中性 box"或"指望外壳模型有内表面"都合理。

### 上位演进（2026-06-15，用户定向）：能力切面注册表 —— 用注册表替代不断扩字段

> **问题意识**：上面的 `terrain_profile` / `interior_skin` / `dressing_assets` / `entrance` 是四个**固定字段**。它把"枚举场景类型"换成了"枚举定制维度"——本质还是预设"这个世界只有地形、内皮、外壳、入口四类可变项"。但开放场景会不断冒出 water / snow / fog / vines / ruins / lighting / road / bridge / river / moat……每来一个就扩一个字段，文档会越来越像枚举，违背开放初衷。

**通用表达：Zone 不再为每种环境维度固定扩字段，而是通过 `aspects` 承载可组合能力。**

```python
Zone {
  id; name; role; volume; enclosure
  aspects: [ZoneAspect]      # ← 新：开放的能力切面列表，承载所有"环境定制维度"
  objects; sub_zones; connectors
}

ZoneAspect {
  capability: str            # 能力名（匹配代码侧注册表）
  params: dict               # 开放参数，由对应生成器解释 + clamp + 中性默认
}
```

代码侧是一张**生成器注册表**，不是 `if/else` 场景分支。**关键：注册表条目不是裸 `name→generator`，而是 `name→{phase, depends_on, generator, effective_params}`**——因为锚定链是强顺序依赖的（地毯必须在蒙古包之后铺、栅栏必须在 shell 之后围），无序 dict 派发会**直接撕断锚定链**，把"从真实足迹派生"打回"从抽象 volume 估"的老穿模 bug。

```python
GENERATOR_MANIFEST = {
  "ground_profile":   {"phase": "GROUND",      "generator": generate_ground_profile,
                       "effective_params": ["type", "amplitude", "frequency", "extent_factor"]},
  "shell_dressing":   {"phase": "SHELL",       "generator": apply_shell_dressing,
                       "effective_params": ["asset_id"]},
  "entrance":         {"phase": "SHELL",       "generator": apply_entrance,
                       "effective_params": ["style"]},
  "interior_surface": {"phase": "INTERIOR",    "generator": generate_interior_surface,
                       "depends_on": ["shell_dressing"],   # 读 shell 真实 AABB 派生地毯尺寸
                       "effective_params": ["floor_material"]},
  "boundary":         {"phase": "BOUNDARY",    "generator": generate_boundary,
                       "effective_params": ["kind"],        # material 暂 accepted-but-ignored
                       "accepted_but_ignored": ["material", "height"]},
  "ground_cover":     {"phase": "DECORATION",  "generator": generate_ground_cover,
                       "effective_params": ["kind", "density"]},
  # objects（家具）不在此表——它走原 compose_scene 布局 LLM，作为 OBJECTS 阶段
}

# 阶段固定顺序（= 锚定链时序 = 渐进化 yield 点 = 失败回滚粒度，三者共用同一组切分）
PHASE_ORDER = ["GROUND", "SHELL", "INTERIOR", "BOUNDARY", "DECORATION", "OBJECTS"]

def dispatch_aspects(zone):
    # 先按 phase 分桶（不按 aspects 列表顺序——那个顺序无意义且会撕断锚定链）
    buckets = {p: [] for p in PHASE_ORDER}
    for aspect in zone.aspects:
        if aspect.capability == "unsupported":
            record_unsupported(aspect.params)   # {requested, reason} → backlog，绝不派发
            continue
        entry = GENERATOR_MANIFEST.get(aspect.capability)
        if entry is None:
            logger.info("未支持的 aspect: %s，跳过（候选新生成器）", aspect.capability)
            continue
        buckets[entry["phase"]].append((aspect, entry))
    # 按 PHASE_ORDER 顺序派发；每个 phase 结束是天然的 yield / snapshot 边界
    for phase in PHASE_ORDER:
        for aspect, entry in buckets[phase]:
            entry["generator"](zone, **aspect.params)
        yield phase   # ← M4 渐进化在此 yield；E4 快照在此存；失败在此回滚到上一 phase
```

**阶段表（PHASE_ORDER）**：

| Phase | 作用 | 示例 capability | 依赖 |
|-------|------|----------------|------|
| `GROUND` | 生成地面 / 平台 | `ground_profile` | — |
| `SHELL` | 放建筑外壳 / 入口 | `shell_dressing`, `entrance` | GROUND（平台定位） |
| `INTERIOR` | 基于 shell/volume 生成内皮 | `interior_surface` | **SHELL（读真实 AABB）** |
| `BOUNDARY` | 基于 anchor 生成围栏 / 边界 | `boundary` | anchor（见下） |
| `DECORATION` | 地表覆盖 / 草 / 花 / 雪 | `ground_cover` | anchor |
| `OBJECTS` | 家具 / 道具摆放 | （走布局 LLM） | INTERIOR |

> **`unsupported` 不进 manifest**：它是 LLM"我想要 manifest 没有的能力"的信号，是日志事件而非可执行能力。派发最前面硬特判 → `record_unsupported` 收口 backlog，**绝不**进 `GENERATOR_MANIFEST`（否则污染"注册表 = 真实能力集"语义）。`entry is None` 是另一回事：capability 在 manifest 内、生成器还没接（过渡态）。
>
> **这不是"注册表细节"，是 M2 质量 + M4 渐进化 + E4 回滚的共同地基**：`PHASE_ORDER` 同时定义了①生成时序（锚定链不断）②渐进化的中断点（每 phase 完 yield，host 分层节奏=peer 到达节奏）③失败降级的回滚粒度（崩在 BOUNDARY 就回滚到 INTERIOR 末的快照）。一处切分，三处复用。

#### 纯室外缺口：`boundary`/`ground_cover` 必须支持非 shell 锚点（否则"通用"对纯室外是假的）

> **当前代码硬伤**（[scene_composer.py:1360](../editor/plugins/AITool/cai_extensions/agent/scene_composer.py#L1360)）：`_generate_fence` 注释明写"ring 半径从 `self._shell_aabb` 派生、必须在 `_place_shells` 之后调"。即**纯室外场景（广场/森林/营地，无建筑）根本长不出栅栏/草**——`_shell_aabb` 是空的。草原能跑通恰恰因为它是"混合"（有蒙古包），不是"纯室外"。教堂验证也测不到这点（教堂是 box、不走 terrain）。**这是个假通用：boundary/ground_cover 绑死了 shell。**

解法：定义通用 **anchor 解析**，BOUNDARY/DECORATION 阶段按 anchor 类型派生参照，不写 `if 蒙古包`：

```python
def resolve_zone_anchor(zone):
    if zone.has_shell_aabb():    return ShellFootprintAnchor(zone)   # 混合：围 shell 真实足迹
    if zone.has_platform():      return PlatformAnchor(zone)          # 纯室外有平台：围中心营地区
    return ZoneVolumeAnchor(zone)                                     # 纯室外/室内：铺 zone 体积范围
```

| 场景 | anchor | 效果 |
|------|--------|------|
| 草原蒙古包（混合） | `ShellFootprintAnchor` | 围蒙古包真实脚印 |
| 纯草原营地（无建筑） | `PlatformAnchor` | 围中心营地区 |
| 废弃广场 | `ZoneVolumeAnchor` | 铺 zone 范围 |
| 火山观测站 | `Shell`/`Platform` | 围平台或建筑 |

不写 `if 蒙古包`，只按 anchor 类型。`_generate_fence`/`_generate_grass` 的几何函数保留，只把"参照谁"从写死 `_shell_aabb` 上移成 `resolve_zone_anchor(zone)`。

#### 但不放飞：半开放 schema（capability 受 manifest 约束，params 开放）

> **硬约束**：**LLM 能决策的上限 = 代码能执行的生成器集合。** 开放 schema ≠ 开放能力。LLM 说"飘雪"但代码没有雪生成器，这条决策落不了地。LLM 也不能凭空造 `{"capability": "dynamic_weather"}` 这种不可执行能力。

所以 demo 期采用**半开放**：
- **`capability` 优先来自 manifest**（受约束，保可靠性、保草原零回归）；
- **`params` 开放**（保通用，材质/高度/密度等由 LLM 自由填，生成器内部 clamp）；
- **未知能力统一收口到 `unsupported`**（保扩展性，且 demo 不因 LLM 幻觉崩）：

```json
{
  "capability": "unsupported",
  "params": { "requested": "water_moat", "reason": "城堡需要护城河" }
}
```

decompose prompt 写清：**优先使用 manifest 中已有 capability；只有当场景强烈需要但 manifest 没覆盖时，才输出 `unsupported` 并说明 requested/reason。** 这些 `unsupported` 日志就是需求驱动的能力扩展 backlog——城堡反复请求 `water_moat`，就去补那个生成器，能力前沿被需求推着长。

#### Demo 期最小能力集（CAPABILITY_MANIFEST）

```python
CAPABILITY_MANIFEST = {
  "ground_profile":   "flat / rolling / dunes / noise 等地形起伏",
  "ground_cover":     "grass / sand / snow / stone 等地表覆盖",
  "boundary":         "fence / wall / hedge 等边界物",
  "interior_surface": "floor / wall / ceiling / openings 内皮",
  "entrance":         "door / curtain / archway 入口",
  "shell_dressing":   "建筑外壳和附属外观件",
}
```

这 6 个足够覆盖：草原蒙古包 / 欧式教堂 / 室内房间 / 室外营地。

#### 与四字段的关系：legacy 表达 + adapter，不推翻重写

> **Zone 的通用表达是 capability aspect 注册表；四字段只是当前能力集的 legacy 表达。** 为降低改动风险，代码先兼容旧字段，落地上先做 adapter，而非一次性删字段。

> **短期/长期边界（防"两边都读"逻辑分叉，关键）**：
> - **短期（兼容期）**：旧字段保留，`normalize_zone_aspects` 在派发前把 legacy 字段一次性转成 aspects。
> - **长期**：**生成链路只消费 `zone.aspects`**，旧字段仅作历史数据/外部输入的兼容入口。
> - **铁律**：生成器函数（`_generate_terrain`/`_generate_interior_floor`/...）**只读 `aspect.params`，绝不再读 `zone.interior_skin`/`zone.terrain_profile`**。adapter 是唯一读 legacy 字段的地方——所有 legacy → aspect 的转换收口在 `normalize_zone_aspects` 一处，下游一律走 aspects。否则实现时一旦有人"既读 aspect 又读旧字段"，决策来源就分叉、两套默认值打架。

| legacy 固定字段 | → capability |
|----------------|-------------|
| `terrain_profile` | `ground_profile` |
| grass / fence（dressing） | `ground_cover` / `boundary` |
| `interior_skin` | `interior_surface` |
| `primary_shell_asset_id` / `dressing_assets` | `shell_dressing` |
| `entrance_hint` | `entrance` |

```python
def normalize_zone_aspects(zone):
    """legacy 字段 → aspects（兼容期双轨，唯一读 legacy 字段处）。

    职责单一：把 zone 上的旧字段转成 zone.aspects 条目（若 aspects 已有同名
    capability 则以 aspects 为准，不覆盖 LLM 直接输出的 aspect）。
    转换完成后，下游生成器只认 zone.aspects，绝不再回头读 interior_skin/
    terrain_profile —— 决策来源唯一，杜绝两套默认值打架。
    """
    # interior_skin → interior_surface aspect
    # terrain_profile → ground_profile aspect
    # dressing 列表 → ground_cover / boundary aspects
    # primary_shell_asset_id / dressing_assets → shell_dressing aspect
    # entrance_hint → entrance aspect
    # ...
```

**一句话**：用能力注册表替代不断扩字段，用 manifest 控制可靠性，用 `unsupported` 日志推动后续能力扩展。

> **落地次序（不立刻大改代码）**：① 本节先定为文档共识；② 下一轮加 `ZoneAspect` + `GENERATORS` 注册表 + `normalize_zone_aspects` adapter（legacy 字段双轨）；③ 逐个把现有生成器（terrain/floor/fence/grass/entrance）接到注册表，每接一个跑草原零回归 + 教堂硬证。**头号风险仍是"改 decompose prompt 动摇草原"**——半开放 schema 下方差更大，故 manifest 给足具体示例锚定 LLM、生成器内部强 clamp、每改一次 prompt 离线重测草原分解 JSON。

任何场景都是一棵 Zone 树，**没有特例**：

```text
草原上的蒙古包：
Zone 草原                enclosure=terrain（开放，无墙无顶）
├─ 栅栏 / 马 / 篝火       objects
└─ Zone 蒙古包内部        enclosure=box（带门洞）+ 蒙古包外壳 dressing
   ├─ 地毯 / 矮桌 / 坐垫   objects
   └─ (可继续嵌套)

欧式教堂：  Zone 教堂  enclosure=box(放大,带门) + 教堂外壳   objects=祭坛/长椅/立柱
现代客厅：  Zone 客厅  enclosure=box                       objects=沙发/茶几/电视
            └─ 退化情况：单 Zone、无 sub_zones、enclosure=box —— 不是"第三类"，是树的最简形态
```

### 关键约束：volume 是事实源，生成的建筑只是「外壳 dressing」

必须诚实承认的物理限制：**文生3D 给的"教堂""蒙古包"是实心外观团块，不是能走进去的空心内部。** 所以：

- 每个 Zone 的**内部体积永远是参数化定义的**（我们自己造的盒子，可控、能开门洞）
- 生成的建筑模型只是**套在体积外面的视觉外壳 (dressing)**，缩放到包住体积、门洞对齐
- 物体永远放进参数化体积里，由 Constraint Solver 摆放；外壳只负责"从外面看像个蒙古包/教堂"

| 场景 | volume（事实源） | enclosure dressing |
|------|----------------|-------------------|
| 客厅 | 房间尺寸 | 合成盒子（= 现在的 `_generate_room_box`）|
| 蒙古包内部 | 内部空间 | 生成的蒙古包外壳，缩放包住体积 + 门洞对齐 |
| 草原 | 一大片区域 | terrain 铺底，无墙无顶 |
| 教堂 | 放大的体积 | 生成的教堂外壳，或合成盒子放大 |

`_generate_room_box` 不被取代，它就是 `enclosure=box` 这一种 dressing 的实现——从"无条件对所有室内生成"变成"Zone 树里某节点声明了 box 才生成"。

### 为什么这才是"开放"的

**开放性不在代码，在分解那一步。** 一个 LLM 把任意 prompt 读成一棵 Zone 树，代码不枚举任何场景类型，只递归实例化 `(volume + enclosure + objects + sub_zones)`。"草原上的蒙古包""赛博朋克酒吧地下室""欧式教堂"对代码是同一件事。我们不写 `if 教堂`，我们让分解器输出 `enclosure=box, scale=large`。

camera 从草原走进蒙古包 = 穿过一个 Zone 边界。没有 Portal / LOD / 动态透明 / 双渲染——蒙古包那面墙几何上留了门洞，就这么简单。**否定一切引擎级渲染技巧，纯几何。**

### 复用已有工程，不另起炉灶

- [multi_scene_parallel_workflow](editor/plugins/AITool/cai_extensions/flows/multi_scene_parallel_workflow/) 的 decompose 已在"把需求拆成子场景"——Zone 树是把那个**扁平的并行子场景**升级成**带空间嵌套 + enclosure 类型的树**。
- `terrain_generation_workflow` 已接入 = `enclosure=terrain` 的现成实现。
- `camera_move` 已验证 = 穿门洞的现成手段。

### 落地分层（Workshop 周期内务实推进）

概念模型立成 general，落地按深度分层，每层都能独立验证：

1. **数据结构定成 Zone**（`enclosure` 是字段不是 if 分支）。初版只支持**单 Zone + box**——把现在的代码"重新表达"成 Zone 退化形态，零功能损失但骨架对了。
2. **支持两层嵌套**（一个外 Zone + 一个内 Zone + 门洞）——足够 demo 蒙古包草原。
3. **shell dressing**（生成的建筑外壳缩放包住体积 + 门洞对齐）、**terrain 与 box 融合**。

工作量集中在：`_detect_scene_indoor` 从二值 bool 升级为"分解出 Zone 树"+ 盒子开洞 + 体积内放置 + 外壳对齐。

### terrain_profile：outdoor zone 的参数化外皮（地形泛化，与 interior_skin 对称）

> **方向（2026-06-14，用户把关）**："从平地到坡度（混合场景/室外场景）要具备泛化性、通用性，不能局限于某一场景，否则违背开放场景初衷。" 所以地形**不是"草原坡度"特例，而是 terrain zone 的参数化属性**——和 indoor zone 的 `interior_skin` 完全对称。

| | indoor zone | outdoor zone |
|---|---|---|
| 事实源 | Volume（盒子） | Volume（一片区域） |
| 外皮 | `interior_skin`（floor/wall/ceiling 参数化） | **`terrain_profile`（高度场参数化）** |
| 开放性在哪 | LLM decompose 读成 skin 取值 | LLM decompose 读成 terrain 取值 |

**通用高度场：一个函数 h(x,z)，type 是参数不是 if 分支**

```python
terrain_profile = {
  "type": "flat" | "rolling" | "terraced" | "dunes" | "noise",
  "amplitude": 起伏高度,
  "frequency": 起伏密度,
  "seed": 随机种子,
}
```

- 草原 → `rolling`（低频缓坡，可叠几道极缓梯田环做层次）
- 沙漠 → `dunes`（正弦波） / 雪山 → `noise` 高幅 / 广场 → `flat`

同一个 mesh 生成器读不同参数。**代码不写 `if 草原`，让 decompose 输出 `type=rolling`**——与处理 `enclosure` 同一思路。小尺度（~20m）以**平滑径向坡**为主，硬阶梯（梯田）在小尺度突兀，故梯田仅作可选缓环叠加。

**通用约束：建筑地基平台（"主建筑在平坦区"的泛化）**

> **任何 terrain Zone 内嵌 shell/box 子 Zone 时，子 Zone 的 footprint + margin 范围强制 h=0（平台），平台外按 profile 起伏。**

- 草原蒙古包：rolling + 中心平台；沙漠帐篷：dunes + 中心平台；雪山木屋：noise + 中心平台；纯室外广场（无 sub_zone）：flat、无平台（没建筑不需地基）。
- 这条规则**一行没写"蒙古包"，但所有"室外套建筑"混合场景自动受益**，且天然保护已修好的 `_place_shells`/布局逻辑——中心平台保平，建筑与家具落点全在 y=0，那些代码一行不改。

**实现决策：自建轻量高度场（不接 terrain_generation_workflow）**

仓库的 `terrain_generation_workflow`（PRESET + heightfield + 植被散布）为 2048m 大世界设计，硬接进 20m 有尺度错配 + 植被穿插 shell 风险。改为**自己写解析高度场 `h(x,z)`**（rolling=径向衰减、terraced=阶梯、dunes=正弦），确定性、可控、小尺度贴合、零穿插。与"参数化、确定性、可控"的 skin 路线一致。草/花归为材质层（15e），与坡度几何分开。

**落地分步（每步独立可验）**

| 步 | 内容 | 解决 |
|----|------|------|
| **15c-ii** | terrain 程序化坡度 mesh（通用 `h(x,z)` + 中心平台通用规则） | 草原太假（几何层） |
| **15d** | wall skin + 壁挂物语义（壁挂类贴 zone 边界、定高、法向朝内） | 兽头标本悬空/朝向 |
| **15e** | 草/花材质层（billboard / 贴图） | 草原太假（材质层） |

> **壁挂物（15d）也走 general**：兽头标本悬空/朝向错，根因是系统不区分"壁挂物 vs 落地物"，且 wall skin 还是空。修法是加**语义**（识别壁挂类：标本/挂画/镜子/壁灯/钟 → 贴 zone volume 边界、~1.5m 高、法向朝房间中心），不是加模型数量。这是 wall skin 的第一个用例，确定性规则、零额外生成。

## 任务 D：锚定链 + 渐进式生成传输（穿模修复 + 多人联机适配）

> **背景（2026-06-14 F5 + 用户/同事确认）**：① 截图暴露各平面比例/对齐错乱（地毯与蒙古包不贴合、壁挂穿模、草原覆盖不一致）；② 用户提出"先地面→蒙古包→地毯→壁挂→家具"的渐进顺序；③ 同事确认 **3D 模型会通过引擎 NetworkSystem 传到其他用户本地**（我先前误判"只传文字"，已纠正）。这三件事收敛成一条主线。

### D-1 穿模根因：各平面各算各的，互不参照

当前 terrain / interior_floor / shell / 壁挂 **每个都从抽象 Volume 独立算尺寸**，还各带一个拍脑袋系数（地毯 INSCRIBE=0.85、蒙古包 WRAP=1.25），从同一 volume 出发乘不同数 → 方地毯（3.4m）和圆蒙古包内壁对不上 → 露天空缝；壁挂贴在抽象方边界 z=hd，真实圆壁是另一半径 → 扎进去/飘出来。**"地毯不贴合"和"壁挂穿模"是同一个 bug：拿抽象 volume 当参照，而非权威物体的真实测量边界。**

### D-2 锚定链（解法，通用、不写 if 场景）

把"放置顺序"改成**权威物体先放、量真实边界，依赖物体从它派生**：

1. **terrain 中心平台 = 唯一地面基准**（h=0，已有）——主建筑地基，蒙古包/地毯/家具都坐平台上，**不需要去贴合坡度**（坡度只在平台外起伏；这步避开了"放到坡度地面"的复杂性）。
2. **shell（蒙古包）先放、贴地、量真实世界 AABB** → 存为"内部基准"。
3. **interior_floor / 壁挂 / 家具从 shell 真实 AABB 派生**（不再用抽象 volume）：地毯尺寸 = 蒙古包足迹 × 内嵌（吃墙厚/圆角），壁挂贴真实内壁半径，家具夹真实边界。
4. 通用：蒙古包/帐篷/木屋同一套，判断在测量、不在 if。

**蒙古包/地毯怎么"落到坡度地面"——B 方案：平台给足 + 脚印夹住（2026-06-14 用户拍板）**

它们**不直接贴坡**，而是落到「坡里挖出的中心平台」（cut-and-fill：先推平一块地基再盖房）。平台是 terrain mesh 自身的一部分（中心 h=0 圆盘，平滑融进外围坡），不是另贴一块。y 方向三者天然对齐到 y=0：terrain 平台钉中心、蒙古包贴地修正抬 min_y→0（15a-fix2）、地毯 y=0.01。

- **为什么不用 A（平台从真实脚印精确派生）**：A 要求蒙古包先于地面放，破坏"地面先长出来→建筑落上→地毯壁挂跟上"的渐进叙事（演示和传输都靠这个顺序），性价比低。
- **B 做法**：① 平台半径**给足**（保证蒙古包一定装得下，宁大勿小）；② 放蒙古包时若真实脚印超出平台 → **夹回平台内**。平台比建筑略大一圈在草原场景里像"营地空地"，自然不违和。
- **遗留（B 也能带）**：平台中心当前钉死原点；将来多建筑/偏移放置时，平台 XZ 应跟建筑走，非永远原点。单建筑居中暂无问题。

**主要工作量 = 重排放置时序**：现状 terrain→地毯（框架阶段）→ shell（很晚，模型生成后）；改成 **shell 先放、量 AABB、再派生地毯/壁挂/家具**。

### D-3 渐进式传输：引擎已替我们做掉大半（纠正后理解）

**真实链路（引擎 Actor 级，非 LANChat 层）**：每个 `Actor` 创建时若未被 `_suppress_network_broadcast` 抑制且有 `_geometry`，自动 `_broadcast_actor_created()`（[actor.py:94-96](../editor/CoronaCore/core/entities/actor.py#L94-L96)）→ `js_call_func("actor-sync-broadcast")` 广播给已连 peer，peer 本地重建。project 外的模型路径（如我们写在 `%TEMP%` 的程序化 OBJ）会被 `_ensure_network_model_path_in_project` 整包复制进 `project/Resource/`（OBJ 连 .mtl + 贴图，[actor.py:266-296](../editor/CoronaCore/core/entities/actor.py#L266-L296)）再传。

**关键推论**：广播挂在**每次 `scene.add_actor()`** 上 → **host 端的分层放置节奏，天然就是 peer 端的渐进到达节奏**。不用手写传输编排/攒批/chunk。锚定链在 host 算好的绝对世界坐标原样广播，peer 重建也对位。**渐进传输几乎是免费的，搭锚定链的车。**

### D-4 待验证/适配点（联机运行时行为，离线验不了）

| 点 | 现状 | 适配 |
|----|------|------|
| 程序化 OBJ 传 peer | `_copy_model_asset_bundle_to_project` 应覆盖 | **F5 联机验**：peer 端有没有地面/草/地毯（不只蒙古包）。没有 → 改"OBJ 直接写 `project/Resource/`"绕开 project 外判断（小改动） |
| `__room_`/`__terrain_` 基础设施 actor 是否广播 | 默认广播（未传 suppress） | 确认这些该传（peer 也要地面），大概率 OK |
| 放置时序 = 广播时序 | terrain→地毯→（晚）shell | 改锚定链顺序，渐进生成 + 穿模修复 + 渐进传输三目标一次对齐 |
| 远端物理 | `_disable_local_physics_for_remote_actor` 已处理 | 无需动 |

### D-5 与 demo 协同的关系

3D 既然真传到 peer，demo 可演**真多人看到同一场景渐进长出**（terrain→蒙古包→地毯→壁挂一层层到达），不止"房主视角"。用户介入（@小D 放大/移动/删除）走菜包 agentic 通道执行 → 改的也是 Actor → 同样广播到 peer。介入联动重派生（蒙古包放大→地毯按新 AABB 重贴）后续可加。collaboration.py 的锁/冲突检测在"房主单人介入"demo 下仍是空转铺路（文档 §任务B P0 已标注）。

> **次要层（与锚定链分开，不混入）**：① 门帘——生成层问题（模型自带红木门），只能在 shell 生成 prompt 加"毡布门帘"引导，不保证；② 花草太假——billboard 纯色固有局限，demo 选项：调淡/关掉/贴图（倾向调淡或关）；③ 草原比例——调 terrain 尺寸/草簇密度，调参非改架构。

### D-6 穿模通用化缺口（锚定链原理通用，但当前实现只在窄场景最稳）

> **诚实定性（2026-06-15）**：锚定链的核心原理"**依赖物从权威物体的真实测量 AABB 派生，而非各自从抽象 volume × 不同系数算**"是通用的，且正好就是 GENERATOR_MANIFEST 里的 `depends_on`（SHELL→INTERIOR→BOUNDARY 顺序的形式化）。**但当前实现只在「单 shell、近圆形、低家具密度」场景下最稳——草原蒙古包恰好全中。** 换场景会暴露四个缺口。这不是否定方案，是把风险说在前面。

| 缺口 | 问题 | 处理 |
|------|------|------|
| **shape-aware fit** | 圆蒙古包和方教堂不能共用同一个 `INSCRIBE`：圆盘地板套方教堂→四角露缝或边缘溢出。圆/方的"内接"数学不同 | 补 `fit_in_footprint(shape, margin)`：地板形状从 shell 足迹派生，不写死圆盘。见下方两档方案 |
| **multi-shell placement** | `_place_shells` 把一切居中放原点（[scene_composer.py:1628](../editor/plugins/AITool/cai_extensions/agent/scene_composer.py#L1628) `set_position([0,0,0])`）→ 两栋建筑压同一点 | shell placement 读 `position_offset`（zone 字段已留），多建筑不压原点 |
| **object collision（家具层）** | 家具互穿、家具扎墙走的是**布局 LLM + 约束求解器**（`compose_scene`），**与锚定链完全两套机制**，注册表/anchor 管不到。草原家具没穿一部分是运气（物体少、空间大）+ 关物理 | E5-b 几何检查兜底（检测≠预防）；根治要求求解器吃真实 AABB，是独立工作流（最远） |
| **anchor drift** | shell AABB 量错会**传染**地毯/壁挂/围栏（它们全从 `_shell_aabb` 派生）——单一基准错则全错，是锚定链"单权威"设计的代价 | AABB 结果进 `generator_log.json`（E8），便于二分排查是基准错还是派生错 |

**shape-aware fit 两档（接口一致，demo 先 Tier 1）**：

- **Tier 1（便宜、铁稳，demo 用）**：用 footprint AABB 长宽比 + 填充率把形状粗分类成 圆/矩形，选 disc/quad 地板。覆盖窄但绝不翻车。
- **Tier 2（principled，北极星）—— floor 套用主建筑底部真实几何（用户 2026-06-15 提出）**：shell 放好后取其网格**底部一段高度（如最低 20%）顶点**→ 投影 XZ 平面 → 求**凹包（concave/alpha shape）**→ 向内缩 margin（吃墙厚）→ 三角化成地板。地板边线天然贴合 shell 内壁、零缝零溢，蒙古包得圆、教堂得矩形/十字形，**全自动不写 `if 形状`**，把 `INSCRIBE` 这个形状敏感常数彻底消解（只剩真正形状无关的 margin）。
  - **坑（必须前置意识）**：① 生成网格是非水密噪声团块，单平面 `section()` 切片可能多环/自交/失败 → **投影+凹包比切片稳**（永远产出一个多边形）；② **凸包会填掉凹陷**（十字教堂带耳堂→凸包鼓成胖团，地板溢到没屋顶处）→ 必须**凹包**，alpha 系数是形状派生常数、不编码场景；③ 噪声网格可能让地板边缘锯齿，反而比干净 disc 难看——这是 Tier 2 不一定优于 Tier 1 的真实风险。
  - **关键边界**：floor 外轮廓可从网格足迹取（它是平的、在最底部），但 [§任务C line 344 铁律](#) "interior_skin 必须参数化、不靠外壳网格内表面" **仍成立**——**墙面/天花的 skin 还是参数化，只有 floor 外轮廓能从足迹派生**。这俩别混。

## 任务 E：产品闭环与演示稳定性（"能生成"→"能稳定交付给用户"）

> **定位（2026-06-15 用户提出）**：Zone + 注册表解决"能生成什么"，锚定链 + phase 解决"按什么顺序生成"，anchor 解决"参照谁生成"。但一个**完整可演示的项目**还差一层——**失败怎么办、用户怎么看、如何回滚、如何复现、多人谁说了算**。这一层与生成质量正交：M2 做完不会让它前进一步，必须单列。
>
> **关键复用**：E2 进度反馈 / E3 中断点 / E4 回滚粒度**全部复用 `PHASE_ORDER` 的 phase 边界**（GROUND→SHELL→INTERIOR→BOUNDARY→DECORATION→OBJECTS）。一处切分，四处复用（生成时序 + 渐进 yield + 进度文案 + 快照粒度）。这是 phase 语义不只是"注册表细节"的第四个理由。

### E1 失败降级（分级，关键对象兜底 / 装饰对象跳过）

按对象在场景中的角色分级降级，不是一刀切：

| 失败类型 | 处理 | 已有/待补 |
|---------|------|----------|
| 关键对象（shell 建筑）生成失败 | 退回 box + 门洞兜底围合 | **已有** `_degrade_failed_shells_to_box`（[scene_composer.py:650](../editor/plugins/AITool/cai_extensions/agent/scene_composer.py#L650)） |
| 关键对象（家具主体）模型失败 | 占位体 / 旧资产（本地模型库）兜底 | 待补：lookup 本地库命中即用 |
| 装饰对象（草/花/栅栏）失败 | 跳过，不阻塞场景 | 待补：phase 内单 aspect try/except |
| terrain 生成失败 | 退回 flat 平地（中性默认） | 待补 |
| unsupported capability | 记日志进 backlog，不阻塞 | **已定** `record_unsupported`（任务 C） |

> 原则：**关键对象失败 → 兜底；装饰对象失败 → 跳过；能力不支持 → 记录**。绝不让单点失败炸掉整场景。

### E2 用户可理解的进度反馈（复用 phase 边界）

渐进式不只是技术顺序，要让用户知道"在干什么、没卡死"。每个 phase 末发一条：

```text
正在生成地形…    (GROUND 完)
正在放置蒙古包…  (SHELL 完)
正在生成内部…    (INTERIOR 完)
正在铺设边界…    (BOUNDARY/DECORATION 完)
正在摆放家具…    (OBJECTS 进行中)
正在等待你确认最终调整…
```

> 文案由 phase → 中性模板生成（"正在生成地形"而非"正在生成草原"——不写场景身份）；具体对象名可填充（"正在放置蒙古包"）但模板通用。

### E3 cancel / pause / resume / finalize（compose 可中断化的产品面）

compose 可中断化（推荐顺序 #6）解决技术中断，E3 是它的用户语义。用户在生成中可能说"停一下""别生成了""保留现在这样""重新生成家具"：

| 指令 | 语义 | 实现挂点 |
|------|------|---------|
| `cancel` | 停止并回滚到本轮开始前 | phase 间检查中断标志 → 回滚到 E4 快照 |
| `pause` / `resume` | 在 phase 边界挂起 / 继续 | yield 点检查 pause 标志 |
| `finalize` | 保留当前已生成的，停止后续 phase | 跳过剩余 PHASE_ORDER |
| 重新生成某层（如家具） | 只重跑指定 phase | 按 phase 粒度重派发 |

### E4 SceneLayout 快照与回滚（复用 phase 粒度）

用户介入后最终调整可能不满意，要能回退。**最小版只存 2 个快照**：

```text
before_final_review   # 用户介入前
after_final_review    # 最终调整后
```

> 进阶：每个 phase 末存一个快照（复用 PHASE_ORDER 粒度）→ "回滚到放家具前"成为可能。但 demo 期 2 个快照够用，phase 级快照后做。回滚 = 重放快照里的 SceneLayout（位置/缩放/旋转），不重新生成模型（模型在 AssetPool，零删除）。

### E5 场景一致性检查（确定性规则，比 VLM 稳）—— 分两层，来源不同不可混

最后一轮不只靠 VLM，加**确定性几何检查**（无 API、可复现、快）。**关键：拆成两层**——基础设施层（terrain/shell/floor/wall/door，走锚定链 + phase 生成）与家具层（走 `compose_scene` 布局 LLM + 约束求解器）来源完全不同，混在一起检查会误判（把求解器的家具穿模算到锚定链头上，或反之）。

```text
E5-a 基础设施层检查（锚定链 / phase 产物）：
  □ shell 是否落在平台上          (shell min_y ≈ 平台 y=0)
  □ interior_surface 是否在 shell footprint 内   (地板边线 ≤ shell 真实足迹，不溢出/不露缝)
  □ boundary 是否围对 anchor      (栅栏环匹配 resolve_zone_anchor 选定的参照)
  □ connector 是否通畅            (门洞未被 shell 壁/家具堵死)

E5-b 家具层检查（compose_scene 布局产物）：
  □ AABB 穿模        (家具间 AABB 相交超阈值)
  □ 挡门            (家具 AABB 与 connector 门洞重叠)
  □ 超出 Zone       (家具中心 / AABB 越过 zone volume)
  □ 放错 Zone       (object.zone_id 与实际位置不符)
  □ 用户锁定物体被移动  (对比快照 + provenance lock_level)
```

> 这些比 VLM 稳定、可单测、不烧 token。VLM 审查作为补充（语义层"看起来对不对"），确定性检查作为底线（几何层"有没有硬伤"）。**分层的价值**：E5-a 失败 → 锚定链/phase/anchor 的 bug（基准量错、派生溢出）；E5-b 失败 → 布局 LLM/求解器的 bug（与注册表无关，见 [D-6 object collision 缺口](#d-6-穿模通用化缺口锚定链原理通用但当前实现只在窄场景最稳)）。两类根因不同、修法不同，检查就该分开报。

### E6 多人同步边界（demo 期：房主单一 authority）

demo 是房主介入、他人观看。规则写死，避免联机歧义：

```text
房主 = 唯一 authority（编辑/介入/Agent 都在房主机器）
其他人 = 只看（接收 actor 广播，本地重建）
远端 actor 不参与本地自动调整（_disable_local_physics_for_remote_actor 已处理）
Agent 只在房主机器运行
锁定信息：demo 期仅房主本地记录即可（owner_id 字段已留，多人锁可见性后做）
```

> 与任务 B（联机 Agent 隔离）、任务 D-3（actor 广播）一致。`network_sync_policy` 的 deferred broadcast 是这层的传输保障（见 demo-fix 记录）。

### E7 本地资产池清理（AssetPool 只增会膨胀）

最小版**只隐藏不物理删**（与 regenerate"资产池零删除"哲学一致）：

```text
rejected 资产：隐藏不展示，不物理删（可恢复）
"清理未使用资产"按钮：手动触发，后做
最大缓存大小 / LRU：更后做
```

> 本地模型库（已建）是 AssetPool 的持久化形态，已有按名复用。清理策略加在库层。

### E8 Prompt / Aspect 可观测性（出问题能定位到层）

每轮生成保存调试快照，定位"错在哪一层"：

```text
original_prompt.txt          # 用户原话
zone_tree.json               # decompose 输出的树
aspects.json                 # 每 zone 的 aspects（含 unsupported）
generator_log.json           # 每 phase 每 generator 的 effective/ignored params
scene_layout_before.json     # 介入前布局
scene_layout_after.json      # 介入后布局
unsupported_aspects.json     # backlog 候选
```

> 排障链：LLM 分解错（zone_tree）→ aspect 参数错（aspects）→ 生成器没支持（unsupported/ignored）→ 布局错（scene_layout）→ 用户改动覆盖错（before/after diff）。每层一个文件，二分定位。

### E9 demo_mode（投稿演示不赌实时 LLM 随机性）

```python
demo_mode = True  # 开关
```

能力：

```text
固定 decompose 输出（缓存的 zone_tree.json，不实时调 LLM）
优先命中本地资产库（不烧混元 token、可反复跑）
关闭高风险 review（VLM 截图死锁风险）
保存可复现日志（E8）
（可选）固定随机种子
```

> demo 录制走**已存盘 `场景1.scene`** 已是 demo_mode 的雏形（静态文件、不实时分解）。E9 是把它产品化成开关。

### E10 验收标准产品化（验用户故事，不只验技术点）

技术验收（任务 C 验证矩阵）之上，加**端到端用户故事验收**：

```text
□ 用户看到场景逐步长出来（E2 进度 + 渐进到达）
□ 用户中途移动/删除/改色（E3 介入）
□ 系统不覆盖用户改动（推荐顺序 #9 用户保护）
□ 最终场景合理（E5 一致性检查通过）
□ 其他人能同步看到（E6 广播）
□ 进入蒙古包后内部合理（INTERIOR phase + 门洞穿越）
□ 失败时不崩（E1 降级）
```

## 一句话总结

> 先关 review 并用 prompt 旋转规则兜底保住主链路；用 **provenance 七字段**（载体先验证，含 batch_id）同时管住 settlement 范围和用户保护；依赖从 **Constraint 关系派生**而非新建图；布局/审查**按批次防抖**；最后一轮只修 Agent、用户物体让位不覆盖；`_engine_lock` 统一收口但当前是铺路项。空间维度走**递归 Zone + 半开放 Aspect 注册表**——`volume` 是事实源、生成的建筑只是外壳 dressing、box 是退化特例；**注册表定义"生成什么"（capability/params），`PHASE_ORDER` 定义"何时生成"（GROUND→SHELL→INTERIOR→BOUNDARY→DECORATION→OBJECTS，同时是锚定链时序 + 渐进 yield 点 + 回滚粒度），`resolve_zone_anchor` 定义"参照谁生成"（Shell/Platform/ZoneVolume，纯室外不绑死 shell）——三者共同保证开放场景、锚定链、渐进式生成不互相打架**；开放性在 LLM 分解那一步而非代码 if 分支；Portal/透明/双渲染一律不做。资产分**资产池/场景布局/用户介入**三层，regenerate 清 Agent 布局但旧模型零删除、可 re-add（demo 不演，后移 P2）。
