# 项目目标锚定版实施总方案

## 1. 总目标

本项目目标不是构建完整产品级平台，但需要达到高质量研究 demo 标准：多人、多 Agent、多阶段场景生成链路必须稳定、可解释、可介入、可审查、可复现。

系统最终应支持如下完整流程：

多人讨论 → 多 Agent 澄清与补充 → GM 提炼模糊意图 → 房主确认 Seed Plan → 分批场景生成 → 用户/Agent 在批次间动态介入 → 几何与 VLM 审查 → 修复与最终调整 → 场景落盘。

所有架构改动必须围绕这个目标展开，不能为了引入新 Agent、新框架或新模型而偏离主链路稳定性。

## 2. 核心矛盾

当前最大问题不是 Agent 能力不足，而是缺少统一控制面和运行时协议，导致以下问题反复出现：

1. 聊天、确认、生成、修改、修复、复盘混在同一条自然语言链路中。
2. 用户补充意图容易被误判为普通 chat，无法进入批次介入。
3. 房主确认后的方案可能再次被 MasterAgent 分类，导致执行失败。
4. 多人冲突、Agent 建议、房主确认之间缺少明确职责边界。
5. 场景生成虽然已支持 progressive compose，但缺少批次级介入、修复和最终调整协议。
6. 室内、室外、混合场景缺少统一结构化约束。
7. 生成模型比例、摆放、接地、穿模、地形适配等问题缺少稳定审查闭环。
8. VLM 审查目前应作为质量审查辅助，但不能替代几何硬约束。
9. 资源生成、下载、后处理和导入缺少统一调度，容易资源溢出。
10. UI 既缺少进度披露，又不能暴露内部 Agent、prompt、工具链细节。

因此，本项目必须从“模型驱动”改为“目标状态机驱动”。

## 3. 目标架构

推荐采用：

InteractionCoordinator + SeedPlan + SceneSession + GenerationScheduler + ScopedMemory + DisclosurePolicy + GM Proposal

整体职责如下：

* InteractionCoordinator：多人链路唯一控制面，负责状态机、消息路由、方案确认、批次介入和事件分发。
* SeedPlan：生成前置协议，承载房主确认后的结构化方案。
* SceneSession：场景生成执行会话，负责 progressive compose、batch boundary、operation log。
* GenerationScheduler：统一资源调度，控制生成、轮询、下载、后处理和导入。
* ScopedMemory：隔离 room、agent、plan、batch、actor 记忆，防止上下文串味。
* DisclosurePolicy：控制 UI 对 host、participant、agent、gm 的信息披露。
* GM Proposal：负责澄清、提案、冲突仲裁和节奏控制，但不直接执行写场景。

## 4. 项目目标拆解

### 4.1 多人模糊意图确认

系统必须支持多人在聊天室中提出模糊、不完整、冲突的想法。

处理原则：

* 普通用户可以自由讨论。
* Agent 可以补充建议，但不能直接触发生成。
* GM 负责把讨论提炼为候选 Seed Plan。
* 房主拥有最终确认权。
* 未确认的 Seed Plan 不能进入生成。

关键机制：

* 所有讨论消息先进入 InteractionCoordinator。
* Coordinator 判断消息属于 chat、intent、clarification、conflict、confirmation 还是 intervention。
* GM 只输出 proposal，不直接执行。
* 房主确认后生成结构化 ConfirmedAction。

禁止行为：

* 普通聊天直接触发生成。
* GM 绕过房主确认生成。
* confirmed action 再次拼成自然语言送入 MasterAgent 分类。

### 4.2 Seed Plan 确认

SeedPlan 是场景生成的唯一合法入口。

SeedPlan 至少包含：

* room_id
* plan_id
* plan_version
* host_id
* participants
* scene_type：indoor / outdoor / mixed
* intent_summary
* style_constraints
* spatial_constraints
* asset_constraints
* placement_constraints
* batch_goals
* review_policy
* conflicts
* confirmed_by
* status

约束：

* confirmed 后不可原地修改。
* 后续用户介入只能形成 intervention 或新 version。
* 所有 batch、actor、review、repair 都必须引用 plan_id 和 plan_version。

### 4.3 场景生成

场景生成必须区分 indoor、outdoor、mixed 三类场景。

室内场景重点：

* 空间边界
* 墙体、地面、入口
* 家具比例
* 动线合理性
* 物体接地
* 室内风格一致性

室外场景重点：

* 地形适配
* 建筑与地面贴合
* 大尺度物体比例
* 边界与路径
* 自然元素分布
* 视线和场景层次

混合场景重点：

* 室内外过渡
* 门厅、庭院、入口区域
* 地形与建筑基座衔接
* 室内风格和室外风格协调
* 物体跨区域摆放一致性

所有场景生成必须遵循：

* 风格一致
* 尊重用户核心意图
* 模型比例合理
* 组装摆放合理
* 不穿模
* 不悬空
* 与地形接触正确
* 入口、动线、边界清晰
* 批次之间语义一致

### 4.4 批次生成与用户介入

生成过程不能视为一次性黑盒执行，而应视为可控批次流程。

推荐批次：

1. layout / boundary
2. main structures
3. objects batch 1
4. objects batch 2
5. decoration / details
6. review and repair
7. final adjustment

用户介入规则：

* 当前批次执行中：介入进入 pending intervention。
* 到达 batch boundary：Coordinator 判断是否应用到下一批。
* 强烈新增需求：进入 next_batch。
* 穿模、比例、摆放问题：进入 repair queue。
* 用户认为执行错方案：暂停后续批次，GM 重新提案。
* 多用户冲突：GM 生成候选仲裁方案，房主确认。
* 最后 1-2 轮明确介入优先级高于早期宽泛偏好。
* 早期介入如果到后期已不合理，最终调整阶段可以修正，但必须记录原因。

InterventionRequest 至少包含：

* intervention_id
* room_id
* plan_id
* session_id
* batch_id
* actor_id
* actor_version
* source_user_id
* intent_type：add / remove / modify / repair / style_adjust / conflict / wrong_plan
* content
* priority
* apply_policy：next_batch / final_adjustment / pause_and_replan / repair_queue

### 4.5 几何与 VLM 审查

质量审查分为硬审查和软审查。

硬审查：

* AABB overlap
* terrain fit
* ground contact
* scale sanity
* boundary violation
* actor collision
* entrance/path blocking

软审查：

* 风格一致性
* 语义缺失
* 用户意图是否满足
* 视觉明显错误
* 场景整体协调性

原则：

* 几何审查优先级高于 VLM。
* VLM 不作为所有生成的唯一阻塞 gate。
* VLM 结果必须结构化写回 ReviewResult。
* ReviewResult 不能只写日志，必须能转化为 repair intervention。
* 真实 VLM 截图审查需要标记为待实机验证。

### 4.6 最终调整

最终调整不是简单收尾，而是解决跨批次冲突和历史介入不一致的问题。

最终调整需要处理：

* 前面批次生成导致的布局不合理。
* 早期用户介入到后期已经不适配。
* 最后 1-2 轮用户强介入需要优先尊重。
* 新增模型的比例、摆放、穿模和地形适配。
* 多用户冲突后的最终折中。
* 风格一致性和整体视觉统一。

最终调整必须输出：

* applied_changes
* ignored_or_modified_interventions
* reason
* remaining_risks
* final_review_status

### 4.7 资源调度

资源调度是本项目稳定性的硬前提。

所有生成任务必须进入 GenerationScheduler，禁止无控制后台线程持续扩散。

调度阶段：

* prepare
* submit
* poll
* download
* postprocess
* import

默认并发建议：

* image submit：4
* model submit：2
* model poll：32
* download：4
* postprocess：1
* import：1

关键要求：

* import 必须单写入。
* submit token 进入 poll 后释放。
* 远端不支持 cancel 时，本地标记 abandoned。
* abandoned 任务的迟到结果不得下载、不得导入、不得创建 actor。
* 生成、下载、导入状态必须可观测。
* 资源队列状态必须能反馈给 UI。

### 4.8 UI 阶段披露

UI 需要缓解用户焦虑，但不能暴露内部链路。

允许披露：

* 当前阶段
* 当前进度
* 是否等待房主确认
* 当前是第几批生成
* 是否处于可介入窗口
* 用户介入是否被吸收
* 是否进入审查或修复
* 是否完成

不允许披露：

* 内部 prompt
* Agent 私有推理
* 工具调用细节
* provider 原始错误
* hidden chain
* raw debug payload
* 内部调度线程信息

推荐 UI 阶段：

* 讨论中
* 方案整理中
* 等待房主确认
* 第 N 批生成中
* 可介入窗口
* 审查中
* 修复中
* 最终调整中
* 完成 / 需确认

用户回执示例：

* “已记录，会在下一批生成中尝试加入。”
* “这个修改涉及当前布局，我会提交给房主确认。”
* “检测到可能穿模，已进入修复检查。”
* “当前方案可能偏离原确认计划，已暂停后续批次并请求重新确认。”

## 5. 实施优先级

### P0：主链路稳定

必须先完成：

1. InteractionCoordinator 控制面。
2. SeedPlan 结构化确认。
3. ConfirmedAction 不再走普通 chat。
4. 多人房间路径强制走 Coordinator。
5. post-compose follow-up 能绑定最近 session。
6. 用户补充、穿模、执行错方案不再被归为普通 chat。

### P1：批次介入和记忆隔离

必须完成：

1. BatchIntervention 协议。
2. SceneSession batch boundary 事件。
3. pending intervention 队列。
4. MemoryScope 隔离。
5. InteractionSummary 结构化注入。
6. 最后 1-2 轮介入优先级规则。

### P2：资源调度与 UI 披露

必须完成：

1. GenerationScheduler MVP。
2. import 单写入。
3. cancel / abandoned / ignore late result。
4. DisclosurePolicy。
5. 用户阶段进度反馈。
6. host / participant / gm / agent 分层视图。

### P3：质量闭环

继续增强：

1. 几何硬审查。
2. VLM 软审查。
3. ReviewResult 转 repair intervention。
4. 最终调整策略。
5. 风格一致性评分。
6. 实验回放与可复现报告。

## 6. AI 执行约束

后续让 AI 改代码时，必须遵守以下锚定原则：

1. 任何代码修改都必须说明服务于哪个项目目标。
2. 不允许只因为技术上优雅而引入新框架。
3. 不允许把 InteractionCoordinator 做成新的大模型 Agent。
4. Coordinator 负责状态机和路由，GM/role agent 负责语义提炼。
5. 不允许 confirmed action 再次走自然语言分类。
6. 不允许多人链路使用全局 scene_id="default"。
7. 不允许把完整聊天室历史塞进生成 prompt。
8. 不允许 UI 暴露内部 prompt、工具链和调度细节。
9. 不允许资源生成绕过 scheduler。
10. 不允许 GM 绕过 host confirmation 直接执行。
11. 不允许 VLM 替代几何硬审查。
12. 不允许为了长期架构一次性重写现有稳定单人 compose 路径。

## 7. 最终结论

本项目的关键不是“让 Agent 更聪明”，而是让复杂人机协作和场景生成过程具备稳定协议。

最终方案应围绕以下主线实施：

InteractionCoordinator 负责控制面；
SeedPlan 负责方案确认；
SceneSession 负责批次生成；
InterventionRequest 负责动态介入；
GenerationScheduler 负责资源调度；
ScopedMemory 负责上下文隔离；
DisclosurePolicy 负责 UI 披露；
GM Proposal 负责澄清、仲裁和节奏控制；
Geometry/VLM Review 负责质量闭环；
FinalAdjustment 负责跨批次一致性修正。

只有先把这条主链路做稳，再考虑更复杂的多 Agent 分工、GraphRAG、自动评价和产品级权限系统。
