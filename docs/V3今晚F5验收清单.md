# V3 今晚 F5 验收清单

日期：2026-06-19

目标：验证 V3 能稳定支撑单人、多 Agent、多人多 Agent、渐进式生成、有限度介入，以及室内/室外/混合场景生成组装。今晚不验证 V4 级大改造，不把完整 BatchOrchestrator、VLM 强审查门、全局布局重排 delta 作为硬门槛。

## 1. 验收前设置

- 分支：`feature/scene-assembly-v3`
- 重新 F5 载入 Python/前端改动。
- VLM 默认关闭；只有单独验证外观审查时由房主打开。
- 首轮资源规模控制在 4-6 个核心物体，避免 8 个全新远程模型全部走混元 3D。

- 不做 CMake/Ninja/native build。

## 2. 单人单 Agent 脚本

### 室外夜市

```text
@商人 我想做一个有点神秘感的室外集市，不要太恐怖，适合几个人逛。
补充：更温暖一点，有灯光和小休息区，不要全是暗黑风。
@商人 简化成最终方案，并确认开始生成。
生成中：新增一个天使雕像。
生成中：再加一只小狗。
@商人 现在生成到哪里了？
完成后：这个边界/栅栏有点奇怪，换成低矮藤蔓灯笼边界。
完成后：查看最终报告。
```

通过标准：

- 状态查询只读，不生成 GM proposal。
- 生成开始后 5 秒内有排队/理解/资源准备类反馈。
- 资源阶段至少出现图片、检索、模型批次之一的用户可见进度。
- “不要太恐怖”不会被当作正向恐怖风格。
- 天使雕像/小狗进入后续批次、追加批，或明确给出服务不可用原因。
- `_terrain_boundary / 栅栏 / 边界` 能定位到系统边界，不误改入口拱门。
- 最终报告列出已吸收、延后、失败、VLM 是否启用。

失败判据：

- 前 10 分钟聊天室完全无生成状态。
- `@商人 现在生成到哪里了` 触发新 SeedPlan 或 GM proposal。
- 完成后要求追加对象时，系统要求用户提供模型路径。
- 边界问题被误改成入口拱门或普通物体。

## 3. 单人多 Agent 脚本

```text
哈喽，大家介绍一下自己。
@长者 我想做一个有点神秘感的室外集市，不要太恐怖，适合几个人逛。
@山贼 我希望它更温暖一点，有灯光和休息区，不要全是暗黑风。
@长者 吸收山贼的方案精华，给出一个简化最终版本。
@长者 按这个方案确认开始。
生成中：后面再加入一个天使雕像。
@GM 现在生成到哪里了？
```

通过标准：

- 长者能吸收山贼方案，不把旧聊天、GM 控制语或工具说明污染成生成物体。
- “确认开始”使用当前 active Coordinator plan，不走 RoleAgent 猜测。
- `@GM 现在生成到哪里了` 只返回当前状态。
- 生成中新增对象进入 pending intervention，不被 RoleAgent 吃掉。

失败判据：

- 生成物体出现“GM 控制台”“模型导入传送门”等聊天/工具元信息污染。
- 已确认 proposal 被重复入队。
- 角色 Agent 声称执行了真实导入，但场景没有对应 actor 且没有失败原因。

## 4. 多人多 Agent 脚本

参与者：房主 1 人，普通用户 1 人以上。

```text
房主：添加夜市验证组。
普通用户：我希望这个地方适合休息，不要太吓人。
房主：@长者 汇总大家意见，给出最终方案。
普通用户：再加一点暖黄灯光。
房主：@GM 确认开始生成。
生成中普通用户：新增一个小狗。
生成中房主：新增一个天使雕像。
普通用户：现在生成到哪里了？
房主：现在生成到哪里了？
```

通过标准：

- 普通用户补充意见进入讨论/介入记录，不直接触发生成。
- 房主确认仍是生成入口。
- 房主看到更完整进度；普通用户不看到内部 worker/job/provider 信息。
- 多人 LAN 同步不再每 2 秒重复广播同一 actor create。
- 大模型同传若仍卡顿，日志应区分文件传输、actor create、场景 snapshot 三类原因。

失败判据：

- 普通用户一句话直接触发生成任务。
- 周期性出现同一 actor 的重复 `Broadcast actor create`。
- 聊天室泄露 `job_id / session_id / provider URL / API key / worker / function name`。

## 5. 室内场景脚本

```text
@小女孩 做一个温暖的小房间，有书桌、床、地毯、台灯和小玩偶，整体可爱但不要太拥挤。
确认开始。
生成中：地毯要更大一点。
完成后：查看报告。
```

通过标准：

- 室内不应派生户外大地形/木栅栏边界。
- 物体比例合理，书桌/床/地毯/台灯有基本空间关系。
- 生成中修改不会覆盖最近用户介入。

## 6. 混合场景脚本

```text
@长者 做一个半室内半室外的茶馆小院，有室内茶桌、门口过渡区、外面小路、灯笼和几盆植物。
确认开始。
生成中：外面再加一个小招牌。
完成后：门口过渡区要更清楚。
```

通过标准：

- 有室内/室外/入口过渡的基本分区。
- 不把全部物体摆成一个室内盒子或一个户外平面。
- 入口/门口/小路等锚点能被后续状态和边界逻辑识别。

## 7. VLM 验证

默认不打开 VLM。单独验证时：

```text
房主打开 VLM 外观检查。
用 3-4 个物体的小场景生成。
完成后查看最终报告。
```

通过标准：

- VLM 开启时报告里说明是否执行、审查对象、建议或失败原因。
- VLM 失败不阻断主生成。
- VLM 不直接改场景，只给建议/最终调整输入。

```powershell
python docs/probes/v3_f5_log_check.py build\examples\engine\RelWithDebInfo\logs\<本轮>_corona.log
```

探针会自动检查：

- 同一 actor/model 是否重复 `Broadcast actor create`。
- `Resource/terrain.obj` 是否重复 `FILE_REQUEST`。
- 生成进度/资源进度是否出现长时间空窗。
- 图片补偿/模型生成是否出现资源心跳。
- 用户可见消息中是否疑似泄露 `job_id/session_id/provider/api_key/worker/runtime_context/prompt`。
- `Mesh 'terrain' / terrain_detail` 是否只是 MeshOpt 分段噪声。

判读：

- `FAIL`：本轮原则上不通过，先看对应日志片段。
- `WARN actor-create`：若只发生在新 peer 加入或显式快照补齐，可接受；若周期性重复，同步未通过。
- `WARN resource-heartbeat/progress-gap`：若资源阶段很快完成，可接受；若用户体感等待明显，则视为披露问题。
- `PASS terrain-meshopt-noise`：说明大量 `Mesh 'terrain'` 是 OBJ 内部 mesh 分段解析，不是地形重复导入。

## 9. 今晚边界

可以接受：

- 远程图片/模型服务慢，但必须持续披露状态。
- VLM 默认关闭。
- 完成后“整体布局不合理”只形成建议或低风险调整，不强求全局自动重排。

不可接受：

- 长时间静默。
- 状态查询触发生成。
- 生成未组装完成却显示成功完成。
- 用户介入完全丢失且没有失败说明。
- 明显泄露内部队列、worker、provider、API、URL、prompt。

## 10. 2026-06-19 追加核验

### 10.1 地形“多次导入”判定

基于 `build/examples/engine/RelWithDebInfo/logs/2026-06-19_02-52-51_corona.log`：

- `Mesh 'terrain' / Mesh 'terrain_detail'` 大量出现，是 `terrain.obj` 内部 mesh/group 分段解析与 MeshOpt DEBUG 输出，不等价于重复导入 actor。
- `ResourceManager Successfully loaded ... terrain.obj` 仅出现 1 次。
- `FILE_REQUEST Resource/terrain.obj` 与对应 `FILE_CHUNK` 仅出现 1 次。
- `terrain / grass / boundary` 后续各重复一次 `Broadcast actor create`，属于旧运行中的场景快照/补齐广播，不是资源重复加载。

当前代码约束：

- host 周期同步必须调用 `broadcastCurrentSceneSnapshot(currentSceneName.value, false, false)`，不得每 2 秒重播 actor create。
- client 显式请求全量快照时仍调用 `broadcastCurrentSceneSnapshot(sceneName, true, true)`，用于新加入 peer 补齐已有场景。
- `editor/Frontend/scripts/test-network-ai-framework-sync.mjs` 和 `editor/plugins/AITool/services/test_native_lanchat_bridge_static.py` 已覆盖该约束。

F5 复验标准：

- 同一 actor 不应周期性重复出现 `Broadcast actor create`。
- 新 peer 加入或显式快照请求后，host 会补发一次必要的 actor create；同一 host 会话内同一 `scene + actor_guid + model_path` 不应被反复补发。
- `terrain.obj` 不应反复触发新的 `FILE_REQUEST`；若反复请求，优先查对端资源缓存/路径映射，而不是生成链路。

2026-06-19 追加修复：

- `Network.vue` 已移除显式快照请求时清空 `snapshotActorCreateKeys` 的逻辑，避免客户端重复请求快照时 host 重发所有 actor create，造成对端看起来“重复导入模型/地形”。
- `test-network-ai-framework-sync.mjs` 已增加静态门禁：`broadcastCurrentSceneSnapshot()` 内不得出现 `snapshotActorCreateKeys.clear()`。
- 新增 actor 仍可正常同步，因为去重 key 包含 `scene + actor_guid + model_path`；停止/重启 session 时 `resetSessionInfo()` 仍会清空去重状态。

说明：

- `verify_ultimate_plan.py` 的语法门禁已覆盖资源链路核心模块：`dispatch.py / retrieve.py / generate.py / formatters.py / helpers.py / progress.py`，防止临验前资源披露或批次链路改动出现语法级漏检。
- `docs/probes/v3_f5_log_check.py` 已可用于每轮 F5 后自动检查日志；旧日志 `2026-06-19_02-52-51_corona.log` 校准结果显示 terrain 文件只请求 1 次，MeshOpt 大量 terrain 日志只是分段噪声。
- 日志探针会输出最终摘要：
  - `F5_READY`：没有 WARN/FAIL，可作为非 native 日志层通过信号。
  - `F5_REVIEW_WARNINGS`：没有阻断项，但需要人工看 WARN；旧日志属于这一类，因为它是修复前日志，存在进度静默和一次性 actor create 补发。
  - `F5_BLOCKED`：存在 FAIL，不建议继续签收，需要先修日志指出的问题。
- `cef-crash` 若出现 `FATAL` / `DCHECK failed` / `render_input_router.cc` / `gpu_channel_manager.cc`，探针会标 `F5_BLOCKED`。这类问题可记录为 `[待 CEF/实机验证]`，但如果当轮 UI 面板已崩溃，不建议把该轮作为通过样本。
- `completion-integrity` 会检查最终完成摘要是否有导入/actor create 证据；若日志已声称“场景组合完成/生成完成”，但摘要中 `导入引擎：0` 或全日志缺少导入证据，会标 `F5_BLOCKED`，避免“未组装完成却显示成功完成”。
- `intervention-visibility` 会检查生成中介入是否有最终处置说明。没有介入标记的轮次仅 WARN；若日志已记录“可介入窗口：已记录/已记录后续补充”等，但最终没有“生成中吸收/已吸收/延后/失败/待补/未处理原因”等说明，会标 `F5_BLOCKED`。
- `verify_ultimate_plan.py` 末尾出现的 401、missing handler、VLM timeout、broken import 等堆栈为测试用例中的故障注入，最终汇总为 `[SUMMARY] All non-native ultimate-plan checks passed.`。
- 本轮未执行 CMake/Ninja/native build，CEF 手势 DCHECK 与真实实机渲染仍标 `[待 F5/实机验证]`。
