## 一、单人方向

### 1.1 初始目标

让用户在"一句话生成场景"的过程中**持续参与**，而不是等待最终结果。四个核心诉求：

- **开放场景**：任意需求拆成 `AssetPool + Zone/Anchor + SceneLayout`，不依赖固定模板（草原蒙古包只是其中一个代表 case）。
- **生成中可介入**：在 phase 边界吸收用户的 move/add/delete/scale/color 修改，后续 batch 不覆盖用户操作。
- **去特殊化**：代码不按场景关键词（if 教堂/if 蒙古包）分支，场景差异由 LLM 输出的 `ZoneAspect.params` 驱动。
- **可靠性**：AABB 防穿模/悬空/挡门（内回路），VLM 做风格一致性审查（外回路 advisory），两层都不阻塞主链路。

### 1.3 关键设计决策

**决策 1：prompt 是 SceneState 的投影，不是事实源**
- 布局、审查、重排时从 `SceneState` 懒重建 prompt，不维护历史 prompt 作为状态。
- 否决了"把每轮 prompt 存下来作为后续对话上下文"的方案——prompt 会越滚越大、且与用户实际编辑不同步。

**决策 2：LLM/VLM/Role Agent 不直接操作引擎**
- 所有引擎写入经过 `EngineWriteGate`。这是铁律。
- 否决了"Agent tool 直接调 `scene.import_model()`"的方案——绕过 gate 会导致多人阶段无法串行化。

**决策 3：先 adapter 后 prompt**
- `normalize_zone_aspects()` 先把 legacy 字段（`TerrainProfile` 等）迁移到 aspects 单轨，prompt 才输出 aspects。
- 代码侧生成器只读 `aspect.params`，绝不回读 `zone.terrain_profile`/`zone.interior_skin`。
- 这是为了杜绝"两套默认值打架"（legacy 字段 vs aspects params 各有一个默认值）。
- 但对 `ground_cover/boundary` 等**无 legacy 字段**的能力，prompt 是唯一来源，"先 adapter"对它们不适用。

**决策 4：去特殊化 ≠ 万能通用**
- 代码不做 `if 教堂/蒙古包/火山口`。场景差异必须来自 LLM 输出的 `ZoneAspect.params`。
- 但接受以下限制：
  - `INSCRIBE=0.96` 保留为安全缩放常数（写了注释，不隐藏）。
  - 不规则 footprint（L 形/多边形）贴合不了（Tier 2 深水）。
  - `dispatch_aspects` phase 分桶派发降级（M2-4→深水，当前硬编码顺序对四个 F5 场景够用）。
- 否决了"一步到位做万能通用 dispatch"的方案：当前能力集只有 6 项，硬编码顺序比通用 dispatch 更可审计，且不被 LLM 输出的 aspects 数组顺序误导。

**决策 5：AABB 是防穿模主路径，VLM 只做 advisory**
- 否决了"用 VLM 看截图判断穿模"的路径——VLM 对几何精度做不到像素级判断，且延迟不可控。
- VLM 定位：风格一致性、语义摆放、朝向、"像不像"。
- VLM 三道保险：import 失败 skip、整体异常 warning+skip、逐 target timeout→skip。不阻塞主链路。

**决策 6：5 面盒子默认打开前墙**
- 不是引擎 feature，是可观测性策略——方便 F5 看到室内家具。
- `open_face` 不交给 LLM 决定，只由环境变量 `CORONA_ROOM_BOX_OPEN_FACE` 控制。
- 否决了"默认封闭盒子 + 做 Portal/动态透明"——不增加渲染复杂度。

### 1.5 已知缺口

| 缺口 | 严重度 | 说明 |
|------|--------|------|
| F5 四场景实机验收 | **P0 阻塞** | 离线绿 ≠ 视觉对。唯一真合并门。四场景：草原蒙古包/欧式教堂/火山口观测站/纯室外集市 |
| LLM decompose 质量不可离线测 | **P0 风险** | `decompose_zone_tree`→快照路径依赖 LLM 真输出，离线测试绕过 LLM。F5 必须确认 snapshot JSON 字段正确 |
| `_select_interior_floor_shape` 宽深比阈值 1.2 | **P1** | 教堂 footprint 若宽深比 <1.2 且 decompose 未输出 `floor_shape=quad`→错套圆地板。F5 必须看 snapshot 的 `floor_shape` 字段 |
| ZoneVolumeAnchor ring 锚声明尺寸非物体 AABB | **P1** | 围栏位置依赖 LLM zone 定尺是否合理，不是代码 bug 但 F5 仍须看相对位置 |
| 不规则 footprint 贴合 | **深水 X-2** | L 形/多边形/连体建筑地板贴合不了，Tier 1 止于圆/方 |
| scatter 几何仍复用 billboard 草簇 | **低优先** | 灌木/岩石/雪斑只通过材质区分，未做独立 mesh |
| `dispatch_aspects` phase 分桶派发 | **深水（M2-4）** | 有意降级，当前硬编码顺序对四场景够用 |

## 二、多人 / 多 Agent 方向

### 2.1 初始目标

2-3 人 LAN 房间内：
- 模型/操作**同传**（一人导入，其他人看到）
- 多人不能同时乱写引擎，**host 是唯一引擎写入者**（host single-writer）
- GM 检测语义冲突（如两人同时要放不同物体到同一位置），**提案而非强制**，房主确认后执行
- 用户 @AI 助手时，正确的 Agent 被触发，不会被历史中其他 @ 污染

### 2.3 关键设计决策

**决策 1：C++ NetworkSystem 是唯一事实源，Python 是语义层**
- 房间/消息/成员/锁/agent roster/intent 全部由 C++ `NetworkSystem` 管理。
- Python 侧（`LanChatAgentOrchestrator`、`LanChatHostActionExecutor`、`LANChatAgentWorker`）只做语义推理：意图分类、GM 提案、房主确认、冲突检测。
- 否决了"恢复旧 Python chat server"——旧 server 与 C++ NetworkSystem 存在房间状态双写风险。

**决策 2：host single-writer + EngineWriteGate**
- guest 操作/指令发给 host，host 通过 `EngineWriteGate.run()` 执行，执行后广播结果。
- host executor 的 `_process_lock` 是 Python 层的串行补充，`EngineWriteGate` 是引擎层的真正锁。
- 否决了"每人各自写引擎 + CRDT merge"——引擎不支持增量 merge，且 CRDT 语义在 3D 场景中不直观。

**决策 3：GM 不替用户拍板，只提案**
- GM 负责：整理冲突 → 提出方案 → 请求确认。
- 房主确认后才入队执行。GM 不直接调用场景工具。
- 否决了"GM 自动裁决"——即使 GM 判断正确，静默执行会破坏多人信任模型。

**决策 4：proposal_id 级别确认（v1.5）**
- 从"看确认词就消费"→显式 `@GM 确认 gm-xxx`。
- 裸"确认/拒绝"保留为 fallback，但标记 `confirmation_mode=bare_text_fallback`。
- 动机：多 proposal、重复冲突或误输确认时不误伤当前 pending proposal。

**决策 5：单人/多人操作路由分离**
- 单人明确增删改移→普通 agent 通道。
- 多人冲突/显式 @GM/多人重大操作→GM proposal。
- `_is_multi_user()` 对 sender_id 与 history 做归一化，避免单人被误判。

**决策 6：typed SceneDelta + actor version/lock 留到深水**
- 当前 GM proposal→确认→host 执行的链路已通，但执行结果是"Python action_payload + LANChat system message 可见性"，不是 C++ typed SceneDelta。
- Actor version/lock/owner、peer delta apply、host identity 可信字段全部留给底层专项。
- 理由：这些需要 C++/Ninja 编译验证，而当前策略是 Python/前端先行。
