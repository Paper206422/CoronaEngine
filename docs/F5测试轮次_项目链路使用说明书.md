## 1. 当前系统目标

当前 F5 轮次验证的是：

```text
开放场景需求
  -> LLM decompose 成 ZoneTree + ZoneAspect
  -> 生成地形 / 主建筑 / 室内 / 边界 / 物体
  -> 渐进导入
  -> 用户可中途移动/介入
  -> AABB 内回路防穿模
  -> VLM 外回路给语义建议
  -> FinalReview 保护最近用户意图
  -> 聊天侧显示阶段进度、介入数、审查结果、F5 snapshot
```

核心验收口径：

```text
不是一次性生成结果，而是用户、AI 助手、Role Agent、GM/静默监听共同维护同一个 SceneState。
```

## 2. F5 前建议环境开关

### 2.1 progressive 默认路径

`SceneComposer.compose()` 当前默认走渐进式路径：

```text
USE_PROGRESSIVE_COMPOSE != 0
```

如需强制回旧链路排查：

```powershell
$env:USE_PROGRESSIVE_COMPOSE="0"
```

正常 F5 不建议关。

### 2.2 VLM 外回路耗时控制

VLM 外审当前是 advisory，不改场景，但会在 compose 返回前同步跑截图/VLM。

默认最多审查 4 个导入 actor：

```powershell
$env:PROGRESSIVE_VLM_MAX_TARGETS="4"
```

如果现场只想先保主链路速度，可以关闭：

```powershell
$env:PROGRESSIVE_VLM_MAX_TARGETS="0"
```

关闭后仍可验证 AABB、渐进导入、用户介入、FinalReview、snapshot。

## 4. 开放场景生成框架

### 4.1 ZoneTree

LLM 不再只生成固定房间，而是把场景拆成 zone：

```text
terrain zone
shell zone
box zone
indoor/outdoor role
volume size/center
parent relationship
style_context
aspects
```

典型草原蒙古包：

```text
terrain: 草原 / outdoor
shell: 蒙古包 / indoor 或 shell
aspects:
  ground_profile
  ground_cover
  boundary
  entrance
  interior_surface
```

典型欧式教堂：

```text
shell: church
aspects:
  interior_surface.floor_shape=quad
  interior_surface.material=stone
不应出现:
  ground_cover
  boundary
  felt curtain / 毡帘偏置
```

纯室外营地/集市：

```text
terrain only
no shell
boundary aspect
radius/margin 控制边界范围
```

### 4.2 ZoneAspect 注册表

当前 manifest 支持：

```text
ground_profile
ground_cover
boundary
interior_surface
entrance
shell_dressing
unsupported
```

原则：

```text
生成代码不按“教堂/蒙古包/火山口”关键词写 if。
场景差异来自 LLM 输出的 aspect params。
未知能力进 unsupported，不崩。
ground_cover / boundary 是 opt-in；没有 aspect 就不生成。
```

### 4.3 地形与建筑贴合

当前重点能力：

```text
TerrainProfile 由 ground_profile aspect 驱动
shell footprint 决定内部地面和边界参照
resolve_zone_anchor 支持：
  ShellAnchor
  PlatformAnchor
  ZoneVolumeAnchor
```

F5 必看：

```text
主建筑不能悬空
主建筑不能埋地
地板/地毯不能露天空缝
围栏不能偏离 shell 中心
纯室外边界不能切穿主体物体
```

## 5. 渐进生成链路

### 5.1 阶段顺序

当前固定阶段：

```text
GROUND
SHELL
INTERIOR
BOUNDARY
OBJECTS
DECORATION
```

文件：

```text
editor/plugins/AITool/cai_extensions/agent/scene_session.py
```

注意：

```text
OBJECTS 在 DECORATION 前。
不要把装饰提前，否则灯、挂饰、摆件可能早于依附对象生成。
```

### 5.2 progressive 主循环

```text
run_progressive_workflow()
  -> _generate_scene_framework()
  -> _place_shells()
  -> SceneSession()
  -> _distribute_assets_to_phases()
  -> incremental_import()
  -> viewport snapshot
  -> SceneDiffTracker
  -> AABB reasonable_provider
  -> FinalReview
  -> VLM advisory review
```

关键文件：

```text
editor/plugins/AITool/cai_extensions/agent/scene_composer_progressive.py
editor/plugins/AITool/cai_extensions/agent/scene_session.py
editor/plugins/AITool/cai_extensions/flows/scene_composition_workflow/incremental_import.py
```

### 5.3 阶段披露 / 进度条

`SceneSession.progressive_compose()` 返回：

```text
progress_timeline
progress_events
phases_run
```

`progress_timeline` 每个实际 phase 有：

```json
{
  "phase": "OBJECTS",
  "status": "start|done",
  "percent": 0,
  "message": "正在摆放物体...",
  "asset_count": 1,
  "imported_count": 1
}
```

用途：

```text
聊天侧显示阶段披露
前端可渲染进度条
F5 可判断生成卡在哪个 phase
```

若聊天侧没出现阶段信息，优先查：

```text
agent_adapter.py::_handle_scene_compose()
scene_composer_progressive.py::run_progressive_workflow()
scene_session.py::progressive_compose()
```

## 6. 用户介入与移动验收

### 6.1 当前最稳移动路径

当前单人移动验收主路径是用户视口拖拽：

```text
用户在编辑器视口移动 actor
  -> phase 边界采集 get_actors() 快照
  -> SceneDiffTracker 对比 baseline
  -> DIFF_MOVED
  -> InterventionOp(OP_MOVE, source="viewport")
  -> SceneSession.drain_interventions()
  -> SceneLayout.mark_user_intervention(lock_level="HARD")
  -> OperationLog 记录 MOVE
```

关键文件：

```text
editor/plugins/AITool/cai_extensions/agent/scene_diff.py
editor/plugins/AITool/cai_extensions/agent/scene_session.py
editor/plugins/AITool/cai_extensions/data_model/layout.py
```

### 6.2 保护规则

用户移动后应被标记：

```text
provenance=USER
touched_by_user=True
intervention_round=current_round
lock_level=HARD
```

后续行为：

```text
最近 1-2 轮用户明确操作 = 强保护
Agent 物体与用户物体冲突 = Agent 让位
FinalReview 不静默覆盖用户近期操作
```

### 6.3 F5 移动验收步骤

建议操作：

```text
1. 发起草原蒙古包营地生成。
2. 等第一批主体/物体出现。
3. 手动拖动桌子、椅子或篝火。
4. 等后续 batch 继续导入。
5. 看聊天侧：
   - operation_count 是否 > 0
   - FinalReview 是否提到保留用户调整
6. 看场景：
   - 被拖动物体是否没被挪回旧位置
   - 后续物体是否尽量围绕新状态摆放
```

### 6.4 AI 助手移动的边界

用户说：

```text
把桌子往右移一点
```

会走：

```text
agent_adapter.py::_handle_edit()
  -> 菜包 agentic 通道
  -> set_actor_transform / remove_model 等工具
```

当前边界：

```text
AI 工具移动主要直接操作引擎 actor。
它不一定显式写入 SceneSession 的 OperationLog。
如果发生在渐进生成期间，下一次 viewport diff 可能捕获到移动。
但这不是最强验收路径。
```

因此本轮验收建议：

```text
主验收 = 用户视口拖拽移动
次验收 = 让 AI 助手移动
```

## 7. AABB + VLM 分层检查

### 7.1 AABB 内回路

负责：

```text
穿模
重叠
挡门
越 Zone
悬空
用户 HARD 物体被移动
Dependent 找不到 anchor
```

触发位置：

```text
SceneSession.progressive_compose()
  -> reasonable_provider()
  -> run_furniture_checks()
  -> check_out_of_zone()
  -> reasonable_map_from_result()
```

关键文件：

```text
editor/plugins/AITool/cai_extensions/agent/consistency_check.py
editor/plugins/AITool/cai_extensions/agent/scene_composer_progressive.py
```

### 7.2 VLM 外回路

负责：

```text
语义合理性
物体朝向
比例建议
整体看起来是否合理
```

触发位置：

```text
run_progressive_workflow()
  -> _run_vlm_advisory_review()
  -> review_models_async()
  -> EngineWriteGate.screenshot()
```

返回字段：

```text
vlm_review
vlm_review_text
vlm_review_skipped
vlm_review_timed_out
```

边界：

```text
VLM 不阻塞主生成。
VLM 不直接改场景。
VLM 不覆盖用户物体。
截图失败/工具缺失/超时都 skip。
```

注意：

```text
review_models_async 名称里有 async，但当前实现是同步 for-loop。
它不阻塞场景生成，但会延迟最终聊天回报。
现场嫌慢可设 PROGRESSIVE_VLM_MAX_TARGETS=0。
```

## 9. 多人 / GM 当前链路

### 9.1 当前架构边界

```text
C++ NetworkSystem
  = 房间、成员、消息、agent roster、trigger、lock、intent 的事实源

Python AITool
  = AI 推理、GM 语义仲裁、Role Agent 回复、场景工具调用

Frontend
  = 展示、用户输入、房主确认入口
```

### 9.2 Agent worker

```text
LANChatAgentWorker
  -> network_pop_lanchat_agent_trigger()
  -> LanChatAgentOrchestrator.handle_trigger()
  -> network_send_agent_reply()
```

关键文件：

```text
editor/plugins/AITool/services/lanchat_agent_worker.py
editor/plugins/AITool/services/lanchat_agent_orchestrator.py
```

### 9.3 GM 当前状态

当前已完成：

```text
GM proposal 结构化
action_payload 保存 source_user_id / target_agent_id / intent_text
房主文字确认或按钮确认后 status=confirmed
worker 可广播 confirmed_gm_action 到 C++ intent 通道
confirmed action 可进入 host-side queue
Host executor 在 EngineWriteGate.run() 内执行，并广播 queued / executing / executed / failed 状态
```

尚未完成：

```text
typed SceneDelta / ActorAdded / ActorMoved / ActorDeleted / ActorUpdated protocol
actor version / expected_version / owner / lock metadata
peer 端 actor delta 强同步应用
C++/Ninja 实机构建验证
```

因此多人当前可验收：

```text
GM 能识别冲突/大操作
GM 能发 proposal
房主能文字确认或按钮确认指定 proposal_id
C++/前端能看到 confirmed_gm_action -> queued_host_action -> executing_host_action -> host_action_executed/failed
```

不能宣称：

```text
多人 typed actor delta 已完整执行并同步。
```

## 12. 聊天侧应出现的回报字段

F5 成功时，聊天侧应看到类似：

```text
场景组合完成
  • 识别物体：N 个
  • 获取模型：N 个
  • 导入引擎：N 个
  • 渐进阶段：INTERIOR / OBJECTS / DECORATION
  • 阶段披露：100% OBJECTS done
  • 最近进度：正在摆放物体（N件）...
  • 捕获用户介入：M 条

最终检查：...
VLM 外审：...
F5 分解快照：...
```

若没有这些字段：

```text
1. 看 USE_PROGRESSIVE_COMPOSE 是否为 0。
2. 看 run_progressive_workflow 是否返回 progressive=True。
3. 看 _handle_scene_compose 是否读取 result 字段。
4. 看聊天返回是否被前端截断。
```

## 13. 常见问题定位表

### 13.1 没有生成开放场景，只出现普通盒子

优先查：

```text
decompose_zone_tree() 是否返回 None
LLM decompose 是否超时/格式错
zone_decompose_snapshot 是否生成
```

修复落点：

```text
scene_composer.py::_ZONE_DECOMPOSE_SYSTEM_PROMPT
scene_composer.py::_llm_decompose()
scene_composer.py::_build_zone_tree()
```

### 13.2 教堂长草 / 长围栏 / 出现毡帘

优先查 snapshot：

```text
是否错误输出 ground_cover
是否错误输出 boundary
entrance.params 是否含 curtain/felt
interior_surface.floor_shape 是否 quad
```

修复落点：

```text
decompose prompt
normalize_zone_aspects()
snapshot probe
```

不要在生成器里写：

```text
if church:
```

### 13.3 草原蒙古包没有围栏

优先查：

```text
snapshot 是否有 boundary aspect
boundary.params 是否合理
_run_original_workflow / framework path 是否调用 _generate_fence
resolve_zone_anchor 是否返回 ShellAnchor
```

修复落点：

```text
scene_composer.py::resolve_zone_anchor()
scene_composer.py::_generate_fence()
decompose prompt boundary 示例
```

### 13.4 纯室外营地没有边界

优先查：

```text
是否无 shell
是否有 boundary aspect
resolve_zone_anchor 是否走 PlatformAnchor/ZoneVolumeAnchor
radius/margin 是否过小或过大
```

修复落点：

```text
resolve_zone_anchor()
boundary.params.radius/margin
Zone volume size
```

### 13.5 用户移动后被覆盖

优先查：

```text
SceneDiffTracker 是否捕获 DIFF_MOVED
operation_count 是否 > 0
OperationLog 是否记录 MOVE
mark_user_intervention 是否设置 intervention_round
list_settleable 是否跳过最近用户物体
```

修复落点：

```text
scene_diff.py
scene_session.py::poll_viewport()
scene_session.py::drain_interventions()
layout.py::mark_user_intervention()
layout.py::protection_level()
```

### 13.6 进度条不动

优先查：

```text
progress_timeline 是否为空
phases_run 是否为空
phase_assets 是否没有分到任何 phase
```

修复落点：

```text
scene_composer_progressive.py::_distribute_assets_to_phases()
scene_session.py::progressive_compose()
agent_adapter.py::_handle_scene_compose()
```

### 13.7 VLM 很慢

原因：

```text
review_models_async 当前同步串行。
默认最多 4 个 actor，每个可能截图多角度。
```

临时处理：

```powershell
$env:PROGRESSIVE_VLM_MAX_TARGETS="0"
```

长期修复：

```text
真正异步化 VLM 外回路
先返回场景完成，再补发 VLM report
```

### 13.8 GM 确认后场景没有同步到 peer

当前 Python v1 已经到：

```text
confirmed_gm_action
-> queued_host_action
-> executing_host_action
-> EngineWriteGate.run()
-> host_action_executed / host_action_failed
```

优先查聊天室 / intent 状态链：

```text
是否出现 confirmed_gm_action
是否出现 queued_host_action
是否出现 executing_host_action
是否出现 host_action_executed 或 host_action_failed
是否出现 Host 执行结果系统消息
```

如果状态链缺失，修复落点：

```text
lanchat_agent_orchestrator.py
lanchat_agent_worker.py
lanchat_host_action_executor.py
```

如果状态链完整，但其他客户端场景没有同步，这是当前底层已知边界：

```text
C++ NetworkSystem typed SceneDelta binding
ActorAdded / ActorMoved / ActorDeleted / ActorUpdated 协议
actor version / owner / lock metadata
peer 端 actor delta 应用
```

### 13.9 GM 提案确认按钮 / proposal_id 验收

当前 v1.6 已支持：

```text
GM proposal 文本包含 gm-xxx
Host 端消息下方显示“确认 / 拒绝”按钮
按钮发送：@GM 确认 gm-xxx / @GM 拒绝 gm-xxx
Orchestrator 校验 proposal_id，不匹配时不消费 pending proposal
裸“确认/拒绝”仍保留为 F5 应急兼容路径
重复确认已处理 proposal_id 时，不再次进入 host queue
trigger 显式携带非 host role 时，Python 拒绝确认
```

F5 检查：

```text
房主能看到按钮，Guest 不显示按钮
点击确认后出现 confirmed_gm_action
点击拒绝后不进入 host action queue
手输错误 gm 编号时，GM 返回编号不匹配
重复点击已确认提案时，GM 返回已处理
若 trigger 携带非 host role，GM 返回只有房主可确认
```

### 13.10 多人 / 多 Agent P0 判定性实验

目标：用最少实验把失败归层，不把 Python、C++、同步协议混成一个问题。

分诊树：

```text
状态链不完整 -> Python worker / orchestrator / executor
状态链完整但场景不同步 -> typed SceneDelta / actor sync
能同步但冲突混乱 -> actor version / lock
确认被误触发 -> host 身份强校验
```

实验 0：基础连通与 agent roster

```text
操作：
1. Host 开房，Guest 加入。
2. Host 添加“学者”。
3. Host 发送：@学者 你好。
判定：
Host/Guest 都看到消息和学者回复 -> 基础链路 OK。
Guest 看不到消息 -> LANChat / C++ history broadcast。
Host 看不到 @学者候选 -> Frontend roster。
消息出现但无 agent 回复 -> L1 trigger 或 Python worker。
第一检查：lanchat.js / lanchat_state.cpp / lanchat_agent_worker.py
```

实验 1：多 agent 不 @ 不误群发

```text
操作：
1. Host 添加“学者”和“山贼”。
2. Guest 不 @，发送：帮我给这个场景提一个建议。
3. Guest 发送：@学者 帮我给这个场景提一个建议。
4. Guest 发送：@山贼 帮我给这个场景提一个建议。
判定：
不 @ 时不能两个 agent 都回复；@学者 只学者回复；@山贼 只山贼回复。
失败指向：L1 route / mention / owner_id/local_peer_id。
第一检查：lanchat_state.cpp::enqueue_agent_triggers_for_message / lanchat_agent_orchestrator.py::_run_role_agent
```

实验 2：GM proposal + Host 按钮确认状态链

```text
操作：
1. Guest 发送：@GM 我想删除桌子，但可能会影响其他人的布局。
2. 等 GM proposal 出现，记录 gm-xxx。
3. Host 点击该消息下方“确认”按钮。
判定：
必须看到：GM 提案 -> confirmed_gm_action -> queued_host_action -> executing_host_action -> host_action_executed/host_action_failed。
无 proposal -> L2 orchestrator。
无按钮 -> Frontend confirmation UI。
无 confirmed_gm_action -> proposal_id confirmation。
无 queued/executing/executed/failed -> L3 worker/executor。
第一检查：lanchat_agent_orchestrator.py / RoomPanel.vue / lanchat_agent_worker.py / lanchat_host_action_executor.py
```

实验 3：状态链完整后 Guest 场景是否同步

```text
操作：
1. Guest 发送：@GM 请把桌子往右移动一点。
2. Host 点击确认。
3. 对比 Host 与 Guest 视口中桌子位置。
判定：
状态链不完整 -> 回实验 2。
Host 没动但 host_action_executed -> L3 语义执行未真实 mutation。
Host 动了 Guest 没动 -> L4 typed SceneDelta / actor sync。
两端位置不同 -> L4 transform delta / coordinate / version。
第一检查：lanchat_host_action_executor.py::_execute_payload / NetworkSystem actor sync / typed SceneDelta 底层记录
```

实验 4：非房主手打确认绕过

```text
操作：
1. Guest 发送：@GM 我要删除桌子。
2. 记录 proposal id gm-xxx。
3. Host 不点按钮。
4. Guest 手打：@GM 确认 gm-xxx。
判定：
理想：GM 拒绝，提示只有房主可确认。
若出现 confirmed_gm_action 或进入 host queue -> host identity verification 缺口。
第一检查：lanchat_agent_orchestrator.py::_consume_confirmation / trigger 是否携带 sender_role/host_user_id
```

实验 5：proposal_id 过期 / 乱序 / 重放

```text
操作：
1. Guest 发送：@GM 我要删除桌子，记录 proposal A。
2. 不确认 A。
3. Guest 发送：@GM 我要把桌子移到门口，记录 proposal B。
4. Host 点击 A 下方确认。
5. Host 点击 B 下方确认。
6. 对已确认的 B 再点一次确认。
判定：
A 应被拒绝为编号不匹配或已过期；B 可确认；重复 B 应提示已处理且不再次入队。
失败指向：proposal lifecycle / replay guard。
第一检查：lanchat_agent_orchestrator.py::_pending_proposal / _processed_proposals
```

实验 6：生成中继续输入

```text
操作：
1. Guest 发送：@学者 生成一个复杂的草原蒙古包营地。
2. 第一条进度消息出现后，Guest 立即发送：@山贼 同时给我一个营地防御建议。
3. Host 发送：我还能继续输入吗。
判定：
输入框不能聚焦 -> UI/CEF 主线程。
能输入不能发送 -> Frontend -> CEF bridge。
本地显示但对端 history 不出现 -> C++ LANChat broadcast。
history 出现但 Python 不处理 -> worker trigger / async worker。
第一检查：lanchat_agent_worker.py::process_once / agent_progress_context.py / NetworkSystem message send path
```

实验 7：同一 actor 双人冲突

```text
操作：
1. 场景中准备一个桌子 actor。
2. Host 用视口拖动桌子。
3. Guest 同时发送：@GM 把桌子移动到门口。
4. Host 立刻确认 GM proposal。
判定：
理想：出现 lock/version 冲突提示或 CommandRejected。
无冲突提示且后到覆盖先到 -> actor version/lock 缺口。
Host/Guest 位置不同 -> L4 sync 缺口。
第一检查：network_lock_object / actor version / expected_version / lock_owner 底层协议记录
```
