## 6. 2026-06-19 P1 小闭环追加：后续批资源计划补全

本轮继续补齐“真正大分批闭环”中的一个关键断点：生成中介入产生的后续资源请求，原本已经能创建 `BatchResourcePlan` 并执行图片 -> 模型流程，但存在两个缺口：

- `BatchResourcePlan` 增加：
  - `retrieval_status`
  - `review_status`
  - `failed_reason`

```text

  -> image_status
  -> retrieval_status
  -> model_status
  -> import_status
  -> review_status
  -> failed_reason
```

## 8. 2026-06-19 P2 追加落地：布局不合理 -> 可执行 reflow delta

本轮补齐“布局不合理，调整一下”这类无明确 actor 的最终调整断点。此前 Coordinator 能把这类消息路由到 `final_adjustment`，但 FinalReview 只能处理明确 actor 的缩放、贴地、VLM correction，导致广义布局诉求无法变成可执行 delta。

已落地：

- `SceneSession.apply_final_adjustments()` 增加低风险 `layout_reflow_delta` 分支。
- 对无明确 actor、但内容包含“布局/摆放/动线/不合理/入口/主街/休息区”等关键词的 final adjustment，执行确定性布局重排。
- 重排只作用于：
  - `layout_status=active`
  - `provenance=AGENT`
  - `touched_by_user=False`
- 不删除、不替换模型，不覆盖用户最近移动过的物体。
- 按物体名称/metadata 推断角色并生成 transform delta：
  - 入口/拱门 -> 前方入口位
  - 摊位 -> 主街两侧
  - 长椅/桌子/休息区 -> 侧边休息区
  - 雕像/喷泉/landmark -> 后方焦点
  - 灯笼/灯光 -> 主路引导
  - 导视牌 -> 入口侧边
- `InteractionCoordinator` 测试增强：完成态“布局不合理”会进入 `final_adjustment_plan.selected`，并保留无 actor 的广义布局请求，供 SceneSession reflow 消费。

## 9. 2026-06-19 P1 追加落地：初始资源准备分批

最新 CodeGraph 复核确认，`SceneComposer.compose()` 仍存在一个“大分批闭环”的关键残留：进入 `run_progressive_workflow()` 前，会先对全部初始物体调用一次 `_run_model_retrieval(items)`。这意味着虽然资源 workflow 已经有长耗时披露，但资源提交本身仍偏一次性全量，和“图片 -> 模型 -> 导入按大批次闭环”的终极目标不完全一致。

本轮先做低风险切口：保留现有 model_retrieval workflow，不改底层图片/模型服务，只在 `SceneComposer` 调用层把初始资产拆成有界资源批次。

已落地：

- 新增 `SceneComposer._run_initial_model_retrieval_batches()`。
- `SceneComposer.compose()` 的 Phase 1 从直接调用 `_run_model_retrieval(items)` 改为调用分批入口。
- 新增环境变量 `CORONA_INITIAL_RESOURCE_BATCH_SIZE`：
  - 默认 `3`，表示初始资源按 3 个物体一批准备。
  - 小于等于 `0` 时回退旧的一次性资源准备路径，便于 F5 对照和应急回滚。
- 每个初始资源批次都会向 `progress_sink` 披露：
  - `资源批次 x/y：正在准备 ...`
  - `资源批次 x/y：完成 ...`
- 每批失败原因会汇总到 `_last_fail_reasons`，避免后续错误报告只保留最后一批失败。
- 初始资源批次会写入结构化快照 `initial_resource_batches`，字段与后续批 `BatchResourcePlan` 对齐：
  - `batch_id`
  - `batch_index / batch_count`
  - `requested_items`
  - `status`
  - `image_status`
  - `retrieval_status`
  - `model_status`
  - `import_status`
  - `review_status`
  - `failed_reason`
  - `resolved_items`
- `compose()` 返回结果会携带 `initial_resource_batches`；即使全部模型获取失败，也会返回这份快照，便于 F5 日志和最终报告定位失败发生在哪个初始资源批次。

## 10. 2026-06-19 P1 追加落地：初始资源批次导入状态回写

上一轮已让初始资源准备产出 `initial_resource_batches`，但批次状态只能证明“资源准备完成”，`import_status` 仍停留在 `pending_import`。这会导致最终报告和 F5 复盘只能看到资源批次，不能判断资源是否真正进入场景。

本轮补齐从资源准备到导入结果的回写：

- `SceneComposer.compose()` 在 progressive/original 组装结果返回后，会调用 `_update_initial_resource_batch_import_status()`。
- 回写数据源包括：
  - `result["imported"]`
  - `result["progress_timeline"][].imported_asset_names`
  - 当 `imported_count >= asset_count` 时，安全使用 `batch_asset_names` 作为整批导入成功证据。
- 这样可以兼容实机里 `imported` 返回 actor_id、不等于原物体名的情况，避免误判导入失败。

状态规则：

- 该资源批次所有 resolved 物体均导入：`import_status=completed`，`status=import_completed`
- 仅部分导入：`import_status=partial`，`status=import_partial`
- 已生成资源但无导入结果：`import_status=failed`，`status=import_failed`
- 未启用导入或资源本身失败：`import_status=skipped`

新增测试覆盖：

- `imported` 只有 actor_id，但 `progress_timeline.imported_asset_names` 有物体名时，仍能正确回写导入状态。
- 全量导入批次 -> `import_completed`
- 部分导入批次 -> `import_partial`
- 无 resolved 资源批次 -> `skipped`

## 11. 2026-06-19 P1 追加落地：资源批次摘要进入最终报告

本轮补齐最终报告侧的安全摘要：

- 不暴露 `job_id / session_id / provider / worker / API key / URL`。

## 12. 2026-06-19 P1 追加落地：初始资源批次进入结构化 progress_timeline

进一步审计后发现：初始资源批次虽然已经有 `progress_sink` 文本和最终报告摘要，但它发生在 `SceneSession.progressive_compose()` 之前，默认不会进入 `progress_timeline`。这意味着上层如果只消费结构化进度事件，仍然可能看不到“前 10 分钟资源准备”的批次状态。

本轮补齐结构化进度链路：

- `SceneComposer._run_initial_model_retrieval_batches()` 在每个初始资源批次开始/完成时，追加 `initial_resource_timeline`。
- 每条事件使用统一形状：
  - `phase=INITIAL_RESOURCE`
  - `status=start/done`
  - `percent`
  - `batch_index / batch_total`
  - `batch_asset_names`
  - `resource_plans`
  - `user_message`
- `SceneSession.PHASE_PROGRESS / _PHASE_DETAILS` 增加 `INITIAL_RESOURCE=准备资源`，避免用户看到泛化的“处理场景”。
