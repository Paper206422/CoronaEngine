## 三、效果不理想的根因分析

### 不是架构问题

V2 的三层 DAG + 差量修正 + 物理沉降 + 挂墙修正 + 默认 scale —— 功能远超 V1。V1 一次全量放置、无物理、无 scale 表，不会比 V2 更好。

### 真正限制在三个层面

**层面 1：模型输入质量（不可控）**

Hunyuan3D 生成的模型：
- 尺寸不可预测（台灯 2m 高、茶几 0.3m 宽）
- 原点位置不统一（有的在底部、有的在中心）
- 默认 scale=[1,1,1] 不反映真实物理尺寸

我们的默认 scale 表只是贴药膏——它假设"名字含台灯的物体应该缩到 0.35"，但不知道模型实际多大。

**层面 2：引擎能力边界**

- `get_world_aabb()` 始终返回 null → 锚点靠估算 → 精度差
- 无 `get_velocity()` / `step_physics()` → 物理沉降只能盲等
- 无墙面几何体 → 挂墙修正只能改 Y，不能贴墙

**层面 3：VLM 本质局限**

- 2D 截图判断 3D 空间 → 视角变化可能被误读
- 评分标准不一致（同一场景有时 62 有时 82）
- 可以发现问题但无法量化（"距离过远" 是 0.5m 还是 2m？）

## 四、后续方向（供讨论）

### 短期（本周可做，低投入高回报）

| 方向 | 投入 | 预期收益 |
|------|------|---------|
| **V1 并发导入** — `_import_actors` 改 ThreadPoolExecutor(3) | 30min | 每层提速 ~1s |
| **确定性回退** — LLM 失败时用等间距网格而非报错 | 30min | 流程不中断 |
| **VLM 问题分类** — 区分 `layout_issue` / `object_issue` | 1h | 模型问题不浪费 retry |
| **默认 scale 扩展** — 从 Hunyuan3D 模型名推测合理 scale | 30min | 更多模型受益 |

### 中期（架构调整）

| 方向 | 说明 |
|------|------|
| **模型预处理** | 生成后读取 glb/obj 的实际 bounding box → 存储为元数据 → 布局时使用真实尺寸 |
| **简化 Tier2** | 当前锚点依赖 AABB（不可用）→ 改为和 Tier1 一样的绝对坐标，或增加"类型感知"的默认偏移 |
| **VLM 审查改采样** | 同场景拍 3 次各 1 角度 → 3 次评审投票 → 提高稳定性 |
| **加入地面参照** | 在 `place_scene_from_items` 中生成简单的地面平面 (Y=0, 灰色大矩形) 和/或后墙 |

### 长期（根本性改善）

| 方向 | 说明 |
|------|------|
| **引擎 AABB 修复** | 排查为什么导入的模型 get_world_aabb() 返回 null |
| **模型尺度标准化** | Hunyuan3D 生成时传尺寸约束, 或生成后按语义类别重缩放 |
| **完整房间模板** | 预设地面/墙面/天花板几何体, 让 LLM 布局时有明确的空间参照 |
