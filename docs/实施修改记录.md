### 任务 D · 锚定链（穿模修复：依赖物从权威物体真实 AABB 派生）

- **日期**：2026-06-14
- **文件**：[scene_composer.py](../editor/plugins/AITool/cai_extensions/agent/scene_composer.py)（`_place_shells` / `_generate_terrain` / `_generate_interior_floor` / `_generate_scene_framework` / `_run_original_workflow` / 后处理 wall_hung）
- **F5 现象**：地毯与蒙古包不贴合（露天空缝）、壁挂兽头穿模、各平面比例/对齐错乱。
- **根因（D-1）**：terrain/interior_floor/shell/壁挂**各自从抽象 Volume 独立算尺寸 + 不同系数**（地毯 INSCRIBE=0.85、蒙古包 WRAP=1.25）→ 方地毯（volume×0.85=3.4m）和圆蒙古包内壁（实际更大）对不上 → 露缝；壁挂贴抽象方边界 z=hd，真实圆壁是另一半径 → 扎进/飘出。"地毯不贴合"和"壁挂穿模"是同一个 bug：拿抽象 volume 当参照而非真实测量边界。
- **改法（D-2 锚定链，5 步）**：
  1. **`_place_shells` 量真实世界 AABB**：放置+缩放+贴地后，读 actor 真实 AABB（x/z 与 y 同源处理 scale），存 `self._shell_aabb[zone_id] = {half_x, half_z}`，作"内部基准"。
  2. **平台给足 + 脚印夹回（B 方案）**：`_generate_terrain` 平台半径系数 1.5→**2.2**（宁大勿小，存 `self._platform_radius`）；`_place_shells` 若 shell 真实脚印超平台 → 按比例缩 scale 夹回（留 5% 余量），重新贴地+量足迹。
  3. **interior_floor 从真实足迹派生**：地毯尺寸 = `shell half×2 × INSCRIBE`（非抽象 volume）；无真实足迹兜底抽象 volume。
  4. **时序重排**：interior_floor 从框架阶段（shell 未放、`_shell_aabb` 空）移到 `_place_shells` 之后 → 正确时序 `terrain → shell → interior_floor`。
  5. **壁挂从真实内壁派生**：后处理 wall_hung 贴墙半径 = `min(shell half_x, half_z) - 0.15`（真实内壁内侧），非抽象 hd。
- **落地机制（B）**：蒙古包/地毯落到"坡里挖出的中心平台"（h=0），不直接贴坡——平台是 terrain mesh 自身的平圆盘，cut-and-fill。y 三者对齐 y=0（terrain 平台 / shell 贴地 min_y→0 / 地毯 y=0.01）。
- **渐进传输（D-3）**：放置时序 = `add_actor` 广播时序，host 分层放置节奏天然成 peer 渐进到达节奏（引擎 NetworkSystem 自动逐 actor 广播，无需手写编排）。
- **离线验证（ALL PASS）**：平台给足（4×4→r=4.4）、脚印夹回（6.0 超 4.4→×0.697→4.18 留余量）、地毯派生（足迹 5.6×5.2→地毯 4.76×4.42，非抽象 3.4）、壁挂内壁（min(2.8,2.6)-0.15=2.45，非抽象 hd=2.5）、无足迹兜底 volume。
- **验证状态**：数据流离线验证全过 + 语法检查；待 F5 看地毯贴合蒙古包内壁、兽头不穿模、蒙古包在平台、peer 端渐进到达。

### M2 步骤 15d · wall_hung 壁挂物语义（修兽头标本悬空/朝向）

- **日期**：2026-06-14
- **文件**：[scene_composer.py](../editor/plugins/AITool/cai_extensions/agent/scene_composer.py)（新增 `_get_placement_type` + 后处理钳制加 wall_hung 分支）
- **F5 现象**：兽头标本悬空、朝向不对，没挂靠在蒙古包侧壁。
- **根因（wall_hung 是死分类）**：`_infer_placement_type`（[asset_metadata.py:56-57](../editor/plugins/AITool/cai_extensions/flows/scene_composition_workflow_v2/asset_metadata.py#L56-L57)）会把"高而扁"的物体算成 `wall_hung`，但**全代码库零消费**（codegraph 搜 `wall_hung` 无结果）。所以兽头分类算对了，下游后处理却只会把所有物体往地面砸（`y=aabb_h/2`），不认 wall_hung → 壁挂物被当落地物 → 悬空/朝向乱。
- **改法**：
  - 新增 `_get_placement_type(actor_name, asset_meta, geo_map)`：读 placement_type（三路匹配兜 A-5 key 撞车），名字兜底壁挂类（标本/兽头/镜子/挂钟/壁灯/鹿角）。
  - 后处理钳制循环加 wall_hung 分支：**贴后墙（z=hd-0.1）、定高 0.55h、法向 +Z 朝内、禁物理、不落地**；多个壁挂物沿 x 错开居中。`continue` 跳过落地钳制/整平/物理消穿模。
- **why 不靠加模型数量**：用户提到"也跟模型生成数量不够有关，没挂靠物"。但壁挂的"挂靠面"在 general 模型里不该是另一个生成模型，而是 zone 的墙（参数化，将来 wall skin）。修法是加**语义**（识别壁挂类→贴墙定高朝内），确定性规则、零额外生成。这是 wall skin 的第一个用例。
- **离线验证**：placement_type 三路匹配、名字兜底（标本/兽头/镜子→wall_hung）、普通家具不误判（矮桌→large_anchor）。
- **验证状态**：语法检查 + 分类逻辑离线验证通过，待 F5 看兽头是否贴墙、朝内、不悬空。

### M2 步骤 15c-ii · 通用坡度地形（terrain_profile 参数化高度场 + 中心平台）

- **日期**：2026-06-14
- **文件**：[zone_tree.py](../editor/plugins/AITool/cai_extensions/data_model/zone_tree.py)（新增 `TerrainProfile` dataclass + `Zone.terrain_profile` 字段）、[__init__.py](../editor/plugins/AITool/cai_extensions/data_model/__init__.py)（导出）、[scene_composer.py](../editor/plugins/AITool/cai_extensions/agent/scene_composer.py)（新增 `_terrain_height` / `_build_terrain_mesh_obj` 纯函数 + `_generate_terrain` 重写）
- **用户需求**：草原太假（无坡度/草/花）；提"梯田坡丘融合 + 主建筑在平坦区"。**把关：地形要泛化/通用，不能局限某一场景，否则违背开放场景初衷。**
- **设计（general，与 interior_skin 对称）**：
  - `TerrainProfile{type, amplitude, frequency, seed}` 是 outdoor zone 的"外皮"，跟 indoor zone 的 `interior_skin` 并列一等字段。
  - 通用高度场 `_terrain_height(x,z,profile,platform_radius)`：**type 是参数不是 if 分支**——rolling/dunes/terraced/noise/flat 走同一函数不同参数，代码不写 if 草原。确定性（seed 派生相位，不用 random）。
  - **通用约束·中心平台**：`r <= platform_radius` 强制 h=0；平台半径从内嵌 shell/box 子 zone 的 footprint × 1.5 算。**"主建筑在平坦区"的泛化**——任何 terrain 套 shell/box 都自动得平台，建筑/地毯/家具落点全在 y=0，**保护已修好的 `_place_shells`/布局逻辑零改动**。平台边缘 smoothstep 过渡防硬棱。
  - `_build_terrain_mesh_obj`：32×32 grid mesh，**世界坐标**（顶点 x,z,h 真实米），Actor `scale=[1,1,1]`（不再缩放——否则 x/z 缩放但 h 不缩放会比例错乱）。
- **实现决策**：自建轻量解析高度场，不接 `terrain_generation_workflow`（为 2048m 大世界设计，硬接进 20m 有尺度错配 + 植被穿插 shell 风险）。
- **decompose**：terrain zone 未声明 profile 时 `_generate_terrain` 默认给 `rolling`（demo 够用）；LLM 输出 type 留 15c-iii。
- **离线验证（ALL PASS）**：① 高度场——平台内(r≤4)恒 h=0、平台外 rolling 有起伏、flat 恒 0（退化=旧平地）；② mesh——1089 顶点/2048 三角、ASCII、中心顶点 y=0、所有面 +Y 朝上；③ 集成——shell 4×4 → 平台 r=3.0、平台内 69 顶点全平、平台外 889 顶点起伏。
- **验证状态**：数据层 + 几何 + 集成三组离线验证全过，待 F5 目测草原有坡度、蒙古包仍在平地、camera 走坡。

### M3-edit · 专用编辑执行路径 + shell 贴地 scale 感知（修移动只回原点、放大丢倍数、穿地）

- **日期**：2026-06-14
- **文件**：[agent_adapter.py](../editor/plugins/AITool/cai_extensions/agent/agent_adapter.py)（新增 `_handle_edit` / `_llm_parse_edit` / `_EDIT_SYSTEM_PROMPT`；edit 路由改走专用路径）、[scene_composer.py](../editor/plugins/AITool/cai_extensions/agent/scene_composer.py)（`_place_shells` 贴地 scale 感知）
- **F5 现象（两个）**：① 蒙古包大小对了但 **y 轴未与地面齐平、穿地**；② AI 助手「把蒙古包y轴向上移动2」**不移动、只返回固定位置 (0,0,0)**。
- **问题 2 根因（链路为 add 设计，move/scale 全断）**：日志 `[SpatialAgent] Solver failed: 'NoneType' object has no attribute 'get'` → `using room center default` → `已移动到 (0,0,0)`。`spatial_agent._build_relation`（[:32](../editor/plugins/AITool/cai_extensions/agent/spatial_agent.py#L32)）对非 add 动作 `return None` → `_solve_with_solver(None)` 在 `None.get("object_id")` 崩 → 回退 room center → 永远 (0,0,0)。且 intent schema **没有字段表达"3倍""y轴+2"等相对量**——"放大3倍"的倍数在解析时就丢了。**SpatialAgent 只为"add 新物体算落点"设计，没有"相对编辑已有物体"的数据通路。**
- **问题 2 改法（专用 edit 路径，绕开坏链）**：新增 `_handle_edit`——① 读引擎真实 actor 列表（含 `__shell_` 前缀、排除 `__room_`/`__interior_` 基础设施）；② LLM 看真实列表 + 原话解析出 `{target（精确 actor 名）, op, move_delta/scale_factor/rotate_delta}`（相对量在此正确提取）；③ 读 actor **当前 transform**，相对量叠加算绝对目标（move +delta / scale ×factor / rotate +deg）；④ `set_actor_transform` 应用（用真实名，无前缀不匹配问题）。完全不走 SpatialAgent/coordinator。
- **问题 1 改法（贴地 scale 感知）**：上次贴地读 `get_aabb()` 当世界 AABB，但它返回**未缩放的局部 AABB**（min_y≈-0.38），蒙古包放大 ~6 倍后真实底部在 -2.3 → 只抬 0.38 仍穿 ~1.9m。新增自适应判据：缩放 >1 且实测 y 跨度远小于"局部尺寸×scale"时，`min_y *= scale_y` 再贴地。
- **验证状态**：两文件语法检查 + 函数/路由分支存在性核对通过；引擎侧（蒙古包齐平地面、"移动2"真移动、"放大3倍"真放大3倍）待 F5。

### M3-intent · 意图三分类（compose / edit / chat）取代关键词路由

- **日期**：2026-06-14
- **文件**：[agent_adapter.py](../editor/plugins/AITool/cai_extensions/agent/agent_adapter.py)（新增 `_INTENT_CLASSIFY_SYSTEM_PROMPT` / `_llm_classify_intent` / `classify_intent`；`__call__` 路由改用三分类）
- **用户提出**：「AI 助手调整完全不执行，应该做一个意图区分，哪些是聊天，哪些指令需要修改调整模型」。
- **根因（关键词当意图分类器的结构性缺陷）**：M3-route 只修了"关键词命中的编辑"，但**关键词匹配本身是错的工具**，F5 日志里两类都翻车：
  - **真编辑漏判 → 掉聊天**：「把矮桌变小」（"变小"在句尾，正则 `变小.{1,20}` 要求后面还有字 → 不匹配）、「蒙古包调大」（"调大"不在词表，只有"调整"）。
  - **抱怨误判 → 进编辑**：「蒙古包没有调整好」（命中"调整"）→ 被当指令塞给 coordinator 解析 = 垃圾。
- **改法**：把原二分类 `_llm_is_compose_intent`（compose / 非compose，导致 edit 和 chat 一起掉进"非compose"→ 全聊天）升级为 LLM 三分类 `classify_intent → compose | edit | chat`：
  - **compose**：从零生成整场景（"生成客厅""蒙古包草原"）→ `_handle_scene(force_compose=True)`。
  - **edit**：改已有物体，有明确操作对象+动作（"放大蒙古包3倍""把矮桌变小""沙发往左移""删掉地毯"）→ `_handle_scene` 单步分流到 coordinator 执行变换。
  - **chat**：闲聊/提问/评价/抱怨，无明确操作（"你好""这个好看吗""蒙古包没调整好"）→ 聊天。
  - 快速路径：明显问候/提问（`_NON_SCENE_PATTERNS`）零延迟直接 chat，不调 LLM。
  - 兜底：LLM 不可用 → 退回关键词（`is_scene_command` 命中再用 `is_compose_request` 分 compose/edit）。
- **验证状态**：语法检查 + 三函数与路由分支存在性核对通过；分类准确率待 F5（"把矮桌变小"→edit、"蒙古包没调整好"→chat、"生成客厅"→compose）。
