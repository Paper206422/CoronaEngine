### §5 · grsai 文生图 400 Bad Request（4/8 物体失败）—— 待查，疑似配置/额度

- **日期**：2026-06-13
- **现象**：第二次运行 8 物体，4 成功（沙发/单椅/落地灯/绿植）4 失败（茶几/电视柜/电视/地毯），失败全是 `httpx.HTTPStatusError: 400 Bad Request for url 'https://grsai.dakka.com.cn/v1/api/generate'`（[client_grsai.py:87](../editor/plugins/AITool/Quasar/ai_modules/image_generate/tools/client_grsai.py#L87)）。
- **关键观察**：补偿重试是**串行**的（约 45s 一个），不是并发打满导致，所以**不是并发超限**。串行请求里仍是有的成功有的 400 → 疑似 ① grsai 账号配额/计费问题，或 ② 特定 prompt 内容被拒。
- **不是我方代码 bug**：grsai key 在 ai_setting.py（skip-worktree，用户管）。
- **待办**：用户侧确认 grsai（grsai_image provider）账号额度/计费状态；若额度正常，则抓一条失败 prompt 看是否内容触发 400。

### A-5 · 修复 asset_metadata key 撞车（六物体共用一个 AABB）

- **日期**：2026-06-13
- **文件**：[asset_metadata.py:97-101](../editor/plugins/AITool/cai_extensions/flows/scene_composition_workflow_v2/asset_metadata.py#L97-L101)（生成侧）、[compose_scene.py:123-138](../editor/plugins/AITool/cai_extensions/flows/scene_composition_workflow/compose_scene.py#L123-L138)（查找侧）
- **改动**：
  - 生成侧 `name = basename(stem)` → `basename(dirname(path))`。混元3D 输出统一为 `.../models/<物体名>/base.glb`，basename 永远是 `base`，导致 `result[name]=meta` 六个物体互相覆盖，只剩最后一个。改用父目录名（物体中文名）作 key。
  - 查找侧 `_match_meta_by_path` 同步改为优先用目录名匹配，保留 basename stem 作兜底。两侧必须一致，否则全查不到。
- **原因**：运行日志实锤 `[metadata] base → ...` 出现 6 次但 `compose_scene: asset_metadata 含 1 个物体 AABB`——六个物体的真实尺寸全被覆盖成最后一个（绿植）。LLM 布局时按 path stem（也是 `base`）查 AABB，六个物体全命中同一条尺寸 → 等于"闭眼估尺寸"，这是 scene.json 里 scale 飘（沙发被放大 2.4 倍）的根因。
- **影响**：对布局质量影响最大的一个 bug，比旋转兜底更根本——AABB 一对，过度缩放/穿插都会缓解。
- **验证状态**：已语法检查（ast.parse 通过），待 F5 运行看日志是否 `含 6 个物体 AABB`

### A-1.6 · 开放式意图路由（LLM 判别，替代关键词清单）

- **日期**：2026-06-12
- **文件**：[agent_adapter.py](../editor/plugins/AITool/cai_extensions/agent/agent_adapter.py)（`_COMPOSE_INTENT_SYSTEM_PROMPT` / `_llm_is_compose_intent` / `is_compose_intent` 新增；`__call__` 路由第 3 步新增）
- **改动**：
  - 新增 `_llm_is_compose_intent(text)`：一次轻量 LLM 调用判断"是否在要求生成完整 3D 场景"，输出 `{"compose": bool}`。复用 [intent_agent.py:75-83](../editor/plugins/AITool/cai_extensions/agent/intent_agent.py#L75-L83) 已验证的 `get_chat_model` + ThreadPoolExecutor + 8s timeout 模式。
  - 新增 `is_compose_intent(text)`：先用 `_NON_SCENE_PATTERNS` 挡掉明显闲聊（零延迟），否则交 LLM；LLM 失败保守返回 False。
  - `__call__` 路由：关键词门（`is_scene_command`）**未命中**时，不再直接掉 chat，先过 `is_compose_intent`（LLM 开放式判别），命中则 `routing → compose (开放式意图)`。
- **原因**：A-1.5 暴露的是关键词清单的**结构性缺陷**，不是漏词。两道门 `_SCENE_COMMAND_PATTERNS` + `is_compose_request` 都是写死的房间类型清单，"赛博朋克酒吧街""海底世界""蒙古包草原"（Task C 目标场景）等清单外描述全部掉进 chat，与"开放式场景生成"的设想从架构上矛盾。改为 LLM 语义判别意图，不再靠词表。
- **延迟权衡**：关键词命中走快路径（零额外 LLM）；只有关键词漏掉的开放式描述才多一次 ~1s LLM 调用；LLM 不可用时退回旧关键词行为（保守，宁漏判不误触发闲聊）。
- **出处说明**：原关键词路由是同事 fox-zq 2026-06-09 commit `77137784 第一波接聊天室测试` 的初版逻辑，非本团队设计。
- **验证状态**：已语法检查（ast.parse 通过），待 F5 运行验证"做个海底世界""来个蒙古包草原"能 `routing → compose`
